# BankInsight 安全政策

## 数据边界

BankInsight 当前只允许使用确定性模拟数据和公开、合规的数据字典。严禁上传或提交：

- 真实银行客户姓名、手机号、证件号、账号、交易流水或征信信息；
- 银行内部经营数据、未公开报表、生产 Schema 或权限配置；
- 能够关联到真实个人、机构或账户的映射表；
- 来源、授权或脱敏方式不明确的数据文件。

`data/processed/bankinsight.db` 是项目初始化脚本生成的小规模模拟数据库，不包含真实客户信息。

## 密钥管理

- DeepSeek API Key 只能保存在本地 `.env` 或受控的部署密钥服务中。
- `.env.example` 只能保留空值或明确占位符。
- 禁止在代码、测试、Notebook、截图、Issue、PR、日志和聊天记录中粘贴真实密钥。
- 禁止提交 GitHub Token、Cookie、浏览器凭据、邮箱密码、私钥和证书。
- 如果密钥疑似泄露，应立即在提供方控制台吊销并重新生成，再清理相关 Git 历史和日志。

## 查询安全

所有生成 SQL 必须经过现有 SQL Safety Checker，仅允许单条只读 `SELECT` 或 CTE。不得绕过允许表、敏感字段、参数绑定、超时、限行和只读数据库连接。

任何新增 Generator 或 Provider 都必须通过相同 `SQLGenerator` Port 接入，并接受同一安全层检查。不得让模型直接访问数据库或执行写操作。

## 漏洞反馈

请不要在公开 Issue 中披露可利用的安全细节、密钥或真实数据。发现问题后，应先通过团队私有沟通渠道联系项目负责人，并提供：

1. 受影响版本和模块；
2. 最小复现步骤；
3. 影响范围；
4. 已采取的临时措施；
5. 建议修复方案。

团队确认修复并完成密钥轮换或数据清理后，再决定是否发布不含敏感细节的变更说明。

## 发布检查

每次发布或推送前至少确认：

- `.env`、`.venv/`、`archive_local/`、日志、Crash Report 和 IDE 文件均被忽略；
- 自动化测试、SQL Safety 测试和数据库外键检查通过；
- 提交内容不含绝对本机路径、个人信息和真实银行数据；
- README 中没有夸大当前能力或承诺尚未实现的安全机制。
