# BankInsight 赛题要求与能力缺口分析

> 本文是对团队《Issues 不足分析》的正式整理版，属于基于当前版本的阶段性审计结果，不代表最终赛题完成度结论。审计事实以当前代码、测试、真实数据结构和已创建的 GitHub Issue 为准；团队原分析材料未在公开仓库重复保存。

## 审计范围与状态定义

本次审计覆盖智能问数主链路、真实数据与评测、多轮交互、图表和解释、权限审计、分析产品能力、性能交付及外部系统适配。状态定义如下：

- **已经实现**：当前代码存在可运行实现，并有测试或运行记录支撑。
- **部分实现**：已有可用能力，但覆盖范围、数据基础或验收指标尚未达到赛题完整要求。
- **仅有架构基础**：已有接口、数据位或占位适配器，但生产行为尚未落地。
- **尚未实现**：当前仓库没有可运行实现，不应在答辩或 README 中描述为现有能力。

## 当前事实基线

当前系统采用轻量 Ports & Adapters 架构，`QueryPipeline` 依次编排 Context Resolver、SQL Generator、SQL Safety、Database Executor、Result Formatter 和 Audit Logger。默认查询路由为 Rule First、LLM Extension；Rule 未命中时由 DeepSeek 完成两阶段语义解析和 SQL 生成，全部 SQL 经过 SQLGlot Safety 后进入只读 SQLite。FastAPI 提供 `/api/v1/query`，Streamlit 展示业务结论、结果表、SQL 和技术 Metadata，87项自动化测试覆盖核心链路。

赛题真实数据已经取得，但当前公开运行版本仍使用 Demo 数据库。真实数据规范化入库、官方200题分类、Gold SQL、标准答案对账和正式评测基准属于第一阶段尚未完成的工作，分别由现有 Issue #4、#5、#6 承接。

## 九类重点缺口

### 多轮对话

`conversation_id` 已贯穿 HTTP DTO 和 `QueryCommand`，但 Pipeline 不读取历史消息，数据库没有会话表，前端 Session State 也只保存当前问题和结果。因此当前只能称为“预留会话标识”，不能称为多轮问数。后续首版只需要会话历史、最近轮次召回、指代消解和连续追问，不应直接引入复杂 Agent。

### 智能图表

内部结果模型已有可选 `chart_hint`，说明接口层面为图表建议预留了位置；当前 Formatter 没有填充该字段，Streamlit 也没有柱状图、折线图等数据驱动组件。后续应先建立“结果形状与业务意图到图表类型”的确定性规则，再考虑由模型补充建议。

### 业务解释

模板 Formatter 能对客户数、余额和交易汇总生成中文摘要，证明解释模块已存在，但覆盖范围只有三个 Demo 问题。正式版本需要基于真实指标定义输出结论、比较基准、异常变化、指标口径和证据，且不得让模型脱离查询结果自由编造原因。

### 细粒度权限

SQL Safety 当前能限制允许表和拒绝字段，Schema 也登记了敏感级别，但这些范围来自查询上下文，不是由登录用户、角色或机构权限动态计算。后续需要把认证身份、角色、机构范围、字段权限和脱敏策略映射到 `UserContext`，Safety 和 Executor 均不能绕过该约束。

### 持久化审计

Pipeline 已在请求开始、成功、拒绝和失败处调用 `AuditLogger`，但 Composition Root 注入的是 `NoOpAuditLogger`。后续可替换为数据库适配器，记录请求、用户、机构、路由、SQL 指纹、耗时、结果规模和错误状态；敏感问题原文与 SQL 是否落盘需要单独制定脱敏策略。

### 下钻、导出与看板

当前页面具有产品化布局和只读经营概览，但没有层级下钻、筛选状态传递、CSV/XLSX 导出、保存视图或看板刷新。该能力依赖真实数据层级、产品方案和权限边界，不应在第一阶段数据模型未冻结时提前开发。

### 性能优化

系统已记录数据库与模型耗时，并具备查询超时和最大行数限制，但尚未在官方200题上建立端到端基准。“三秒”只能作为待验证服务目标，必须区分 Rule、本地数据库和外部 LLM 路径，报告 P50、P95、失败率和超时率，不能用单次演示代替性能结论。

### 国产化部署

当前验证环境是 macOS 与本地 Python 虚拟环境，没有 Dockerfile、Linux 部署验证或国产软硬件兼容报告。后续应先完成可重复容器交付和标准 Linux 验证，再根据比赛平台实际环境补充国产数据库、操作系统或模型网关适配，不应提前宣称完成国产化。

### 标准化 API 对接

FastAPI v1 契约和 Ports 已经形成内部替换基础，但尚未接入真实银行数据中台、身份平台或模型网关。第二阶段可以提供接口规范、Mock Server 和一个适配器示例，证明边界可替换；“真实银行中台接入”只有在获得明确系统和联调证据后才能成立。

## 阶段结论

当前六个 Issue 仍是必要且有效的第一阶段任务，负责把现有 Demo 迁移到真实赛题数据并建立可验证业务基础。第一阶段完成不代表全部赛题能力已经交付，但没有规范化 Real 数据、官方200题体系、Gold SQL 和基线评测，第二阶段任何高级功能都缺少可靠输入与验收依据。

第二阶段应按[第二阶段技术路线图](第二阶段技术路线图.md)依次推进。五份候选任务只保存在 `docs/issues/phase2/`，在第一阶段完成、项目负责人确认优先级和接口边界后，才转为正式 GitHub Issue。
