# 02号任务：产品功能与交互设计

> 对应 GitHub Issue：#2“产品功能与交互设计”
> 负责人：两名产品队友共同负责
> 当前状态：任务说明已确认，等待基于官方题库开展产品能力归纳与原型设计。

## 一、根据官方题库整理产品功能

两位负责人共同阅读《问题答案清单》，逐题判断问题类型、完成查询所需的条件、后端需要返回的数据以及前端适合采用的展示方式，将200道题归纳为可复用的产品能力。

首轮分类包括：

- 单值查询；
- 跨期比较；
- 机构排名；
- 趋势分析；
- 多机构对比；
- 衍生指标计算；
- 综合分析。

每道题先记录问题编号和初步类型，再从题目中提取机构、指标、时间、比较基准、排序范围等必要条件。随后结合官方答案，确认后端需要返回单个数值、当前值与基准值、排名列表、时间序列、多机构明细、计算过程或综合指标集合，并为前端匹配结论、指标数值、表格或图表等展示方式。

两位负责人的首轮分工如下：

- 第一位负责人整理单值查询、跨期比较、机构排名和趋势分析；
- 第二位负责人整理多机构对比、衍生指标计算和综合分析；
- 两人共同合并结果，统一分类名称、必要条件和展示方式，并处理重复或跨类型问题。

最终形成“产品能力与展示方式表”，至少包含：

| 字段 | 内容 |
|---|---|
| 问题类型 | 单值、比较、排名、趋势、多机构、衍生指标或综合分析 |
| 代表问题编号 | 可回溯到官方题库的编号 |
| 必要条件 | 机构、指标、日期、时间范围、比较基准等 |
| 后端返回内容 | 指标值、基准值、变化量、排名、时间序列或明细 |
| 前端展示方式 | 业务结论、指标数值、表格、柱状图、折线图或组合展示 |
| 审核状态 | 待整理、待合并、已确认 |
| 备注 | 跨类型归属、特殊展示要求或依赖说明 |

该表是智能问数页面和分析工作台原型的共同输入，也为3号后续确定接口字段和前端结果模型提供产品依据。

## 二、设计智能问数页面

两位负责人根据“产品能力与展示方式表”确定不同查询结果的页面呈现方式：

- 单值问题展示核心结论和指标数值；
- 跨期比较展示当前值、基准值和变化结果；
- 排名问题展示排名表，并根据需要配柱状图；
- 趋势问题展示折线图和明细数据；
- 多机构对比展示机构对比表和适用图表；
- 衍生指标展示结果、公式口径和所用基础指标；
- 综合分析展示业务结论、关键指标和数据明细。

页面原型需要覆盖以下状态，并明确每种状态显示的信息和用户下一步操作：

| 页面状态 | 展示内容 | 用户下一步操作 |
|---|---|---|
| 初始 | 问题输入、必要提示和可用示例 | 输入或选择问题 |
| 查询中 | 当前处理状态和加载反馈 | 等待结果 |
| 正常结果 | 结论、关键数据、表格或图表 | 查看详情、保存结果或继续提问 |
| 条件不足 | 已识别内容和需要补充的条件 | 补充机构、指标或时间 |
| 无数据 | 查询条件与无数据说明 | 修改条件后重新查询 |
| 查询失败 | 可理解的错误说明和恢复入口 | 重试或调整问题 |

最终形成能够直接指导前端开发的智能问数页面线框图或交互原型。原型需要标注主要区域、信息层级、状态变化、操作入口和结果组件，不要求在本阶段实现页面代码。

## 三、设计分析工作台

分析工作台用于承接用户保存的查询结果，并支持后续查看、备注和连续分析。当前阶段重点设计四项功能：

1. **保存查询结果**：将当前问题、系统结论、关键数据、图表、查询时间和必要的技术标识保存为一条分析记录；
2. **查看历史分析**：按时间查看已保存记录，并能够打开某次分析的完整内容；
3. **添加人工备注**：用户可以为保存的分析补充业务判断、处理意见或后续事项；
4. **继续提问**：用户从已有记录发起后续问题，并在页面中保留当前分析上下文的提示。

分析记录原型至少展示：

- 原始问题；
- 系统结论；
- 关键数据；
- 图表；
- 查询时间；
- 人工备注；
- 继续提问入口。

两位负责人共同绘制分析工作台页面原型和基本操作流程，说明用户如何从智能问数页面保存结果、如何在历史列表中找到记录、如何添加备注，以及如何从已有结果继续提问。

本任务最终交付三项成果：

1. **产品能力与展示方式表**；
2. **智能问数页面原型**；
3. **分析工作台页面原型**。

三项成果完成合并、去重和组内确认后，交给3号任务用于接口返回结构、前端内部结果模型和页面实现设计。
