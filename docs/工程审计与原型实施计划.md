# BankInsight 工程审计与原型实施计划

> 审计日期：2026-07-12
> 审计范围：第一阶段“工程审计与接口冻结”
> 结论边界：本文描述的是当前代码的真实状态和后续实施建议，不表示纵向原型已经实现。

## 一、审计结论

当前工程是一个“设计较完整、运行链路未闭合”的初始骨架。

- **能够真实运行的部分**：SQLite 建表脚本、3 个数据库结构测试、FastAPI 应用启动、`GET /health`、SQLGlot 基础安全校验。
- **只有接口或编排骨架的部分**：问题理解、Schema 召回、查询规划、SQL 生成、数据库执行、结果分析、Query Pipeline。
- **尚不存在的部分**：真实 LLM 调用、独立 LLM Provider、持久化 SQLite 数据库、数据装载、数据库执行器、结果格式器、网页前端、权限与审计实现、查询接口端到端测试。
- **当前可运行边界**：安装依赖后可以启动后端并访问 `/health`；`POST /api/v1/ask` 必然返回 HTTP 500，因为 Pipeline 依赖注入函数是显式占位实现。

因此，现阶段不能把工程描述为“可用的银行智能问数系统”，更准确的说法是：**已有数据模型、配置、API 外壳和安全检查基础，但尚未形成从提问到查询结果的纵向闭环。**

2. **SQLite DDL**：建表顺序、主外键、检查约束和索引可由 SQLite 执行。
3. **基础安全逻辑**：SQL AST 检查能够识别只读查询、危险写操作、多语句、未知表和敏感列。
4. **FastAPI 外壳**：应用对象、路由注册、OpenAPI 和健康检查可工作。
5. **内部领域模型**：已有较丰富的意图、查询计划、安全结果和证据模型，可作为后续内部调试结构的参考。

## 七、当前无法形成纵向闭环的原因

1. **主链路没有实例**：Pipeline 依赖的解析器、召回器、生成器、执行器和分析器全部没有具体实现，API 也没有可注入的 Pipeline。
2. **没有可查询的数据底座**：DDL 能建表，但没有 SQLite 数据库、初始化命令或最小演示数据。
3. **外部契约不稳定**：当前请求缺少 `user_id`，响应强制暴露内部 Intent/Plan/Safety 结构；错误没有统一 JSON 格式，前端难以独立开发。

此外，缺少启动文档、依赖锁定和端到端测试，会让6人协作时出现“每个人电脑运行方式不同”和“接口各自理解”的问题。

## 八、第一版纵向原型所需的最小组件

下一阶段只需补齐以下六个最小组件，即可支持2—3个固定问题：

1. **数据库初始化器**：执行 `schema.sql`，写入少量确定性演示数据，生成 `data/processed/bankinsight.db`。
2. **DatabaseExecutor**：只读连接 SQLite，参数绑定，强制最大行数，返回统一 `QueryResult`。
3. **SQLSafetyChecker 适配器**：复用现有 `SqlSafetyValidator`，对外满足冻结接口。
4. **LLMProvider / 规则回退**：先实现预置问题到 SQL 的规则 Provider；LLM Provider 接口存在但可暂不联网。
5. **最小 QueryPipeline**：只编排“生成—校验—执行—格式化”，捕获已知错误并返回统一响应。
6. **最小网页**：Streamlit 页面仅包含提问框、SQL、表格、摘要和警告，统一调用后端 API。

首版不要加入 RAG、向量库、多 Agent、复杂权限、自动修复或复杂图表。

## 十一、后续三个阶段实施顺序

### 第二阶段：最小纵向原型

目标：2—3个固定问题端到端可用。

- 建库并写入最小演示数据；
- 实现规则 Provider、DatabaseExecutor、安全适配器、结果格式器；
- 按冻结契约组装最小 Pipeline；
- 修复 `/api/v1/query`，增加 API 测试；
- 暂不开发复杂网页，只用 API/Swagger 验证闭环。

验收：固定问题返回 SQL、二维表格、摘要；危险 SQL 被拒绝；错误均为结构化 JSON。

### 第三阶段：可替换组件与简单网页

目标：不同成员能够基于固定接口并行开发。

- 接入真实 LLM Provider，同时保留规则回退；
- 实现 Schema/Metric Context Builder；
- 增加 Streamlit 网页，仅调用后端 API；
- 增加请求审计 No-op/文件实现；
- 扩充到15个问题并建立 Gold SQL 测试。

验收：Provider、数据库、安全和前端可独立替换；15个问题有可重复测试结果。

### 第四阶段：评测、产品化与交付

目标：形成竞赛可演示版本。

- 增加 Schema Linking、结果解释、图表和推荐追问；
- 评测 SQL 执行准确率、安全拦截率和时延；
- 完善日志、基础权限、配置管理；
- 再进行 Docker 化和答辩材料制作。

验收：三个答辩案例稳定复现，指标可量化，容器可启动。

## 十二、第一阶段冻结结论

本阶段没有重写 Pipeline、没有接入 LLM、没有生成数据、没有开发网页，也没有删除原文件。下一阶段应首先实现 **SQLite 初始化 + DatabaseExecutor + 规则 Provider + 最小 Pipeline**，并严格遵循[接口契约](接口契约.md)。
