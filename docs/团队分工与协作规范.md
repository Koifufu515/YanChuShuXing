# 言出数行——银行智能问数与协同分析系统 团队分工与协作规范

本文档用于固定六人团队的长期职责边界、交付关系和 GitHub 协作流程。每位成员按工程职责认领任务，不按专业背景笼统分工。具体人员由项目负责人在线上 Issue 中分配。

## 一、协作原则

1. `main` 始终保持可运行，普通成员不得直接向 `main` 提交代码。
2. 每项工作必须先有 Issue，再从最新 `main` 创建独立分支；一个分支和一个 Pull Request 只处理一个 Issue。
3. 产品、前端、数据、指标和测试通过文档或冻结接口协作，避免多人长期编辑同一文件。
4. Query Pipeline、Ports、Composition Root、数据库结构和公共指标配置属于核心变更，必须由项目负责人审核。
5. Pull Request 必须提供验收证据；“本地能运行”不能替代测试结果、截图、查询结果或数据校验记录。
6. 任何真实客户数据、银行内部数据、API Key、Token、Cookie、`.env` 和个人凭据均不得提交。

### 执行顺序

当前#1至#6负责正式数据、官方问题、产品、前端和评测基座。#4、#5、#6先冻结数据、语义和评测契约，#2、#3随后完成产品与前端迁移，#1负责最终集成。第一轮完成不等于多轮、复杂图表、权限、审计、下钻、性能和部署已经交付；后续能力只有在正式数据和回归基准稳定后，才通过新的独立Issue启动。阶段目标、输入、交付物和验收标准见根目录[任务计划](../task_plan.md)。

## 五、标准协作流程

1. **认领 Issue**：在 Issue 留言说明认领，并由项目负责人设置负责人和状态。
2. **同步主分支**：拉取最新 `main`，确认本地测试基线正常。
3. **创建分支**：按 `类型/简短任务名` 命名，不在同一分支处理多个任务。
4. **小步提交**：提交信息说明实际成果，不提交缓存、日志、密钥、个人材料或本地数据库临时副本。
5. **创建 PR**：关联 Issue，填写模板，附测试证据、截图或数据校验结果，并说明接口及数据库影响。
6. **交叉审核**：先由矩阵中的相关角色审核，再由项目负责人处理核心路径或最终集成。
7. **修正与复验**：作者处理意见并重新给出验证结果；审核人确认问题关闭。
8. **合并与同步**：项目负责人使用 GitHub 合并，作者再同步最新 `main`；已合并分支可在远端删除。

## 七、冲突处理

1. 提交 PR 前执行 `git fetch origin`，再在自己的分支合并 `origin/main`。
2. 初学成员优先使用普通 merge 同步，不使用强制推送，不自行改写共享分支历史。
3. 只解决自己理解且负责范围内的冲突；涉及 Pipeline、Ports、公共配置或同一指标口径时，立即请求项目负责人和相关模块负责人共同处理。
4. 不得通过删除对方代码、`git reset --hard` 或强制推送“解决”冲突。

## 八、主分支保护建议

项目负责人应在 GitHub 仓库的 **Settings → Rules → Rulesets**（或 Branches）为 `main` 配置：

1. 合并前必须通过 Pull Request；
2. 至少一名审核者批准；
3. 新提交后撤销过期批准；
4. 要求对话全部解决；
5. 禁止强制推送；
6. 禁止删除 `main`；
7. 团队成员和测试稳定后，启用必须通过状态检查；
8. GitHub 方案支持时，启用 CODEOWNERS 审核。

启用“必须由 CODEOWNERS 审核”前，应先确认至少一名非 PR 作者具备审核权限，避免只有一个仓库管理员时无法完成批准。保护规则是否已生效必须以 GitHub 设置页或 API 返回为准，不能仅凭文档判断。

## 九、完成定义

一个任务只有同时满足以下条件才算完成：Issue 交付物齐全、验收标准逐项通过、测试和文档同步、无敏感信息、所需审核完成、PR 已合并。代码写完但没有证据或未合并，只能标记为“待验收”。
