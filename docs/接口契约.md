# 言出数行——银行智能问数与协同分析系统 当前接口契约

> 契约版本：HTTP API v1 / 内部接口 v2
> 对应阶段：Demo 技术基线 / 正式数据迁移
> 校准日期：2026-07-16
> 状态：当前稳定契约。正式数据迁移原则上不改变 HTTP 顶层结构和内部 Port 方法。

## 一、契约边界

“言出数行”对外提供一个同步查询接口，对内通过轻量 Ports & Adapters 组织能力。外部 DTO 使用 Pydantic，内部命令、结果和 Port 参数使用框架无关的 dataclass。

当前查询链路为：

```text
HTTP QueryRequestDTO
  -> QueryCommand
  -> ContextResolver
  -> SQLGenerator（Rule / LLM / Rule First Hybrid）
  -> SQLSafetyChecker
  -> DatabaseExecutor
  -> ResultFormatter
  -> QueryOutcome
  -> HTTP QueryResponseDTO
```

Pipeline 只负责编排，不读取 YAML、不调用模型 SDK、不解析 SQL AST、不打开数据库连接，也不决定 HTTP 状态码。

## 二、外部 HTTP API

### 2.1 查询接口

```http
POST /api/v1/query
Content-Type: application/json
```

请求结构：

```json
{
  "question": "查询客户C001的账户余额",
  "user_id": "demo_user",
  "conversation_id": "demo_session"
}
```

| 字段 | 类型 | 必填 | 约束 |
|---|---|---:|---|
| `question` | string | 是 | 去除首尾空格后2到500个字符 |
| `user_id` | string | 是 | 1到64个字符 |
| `conversation_id` | string/null | 否 | 最长128个字符 |

当前 Demo Streamlit 客户端始终传入 `user_id=demo_user`。API DTO 本身要求 `user_id` 存在，没有该字段时返回统一的422响应。正式阶段如增加用户身份或机构范围，应通过认证上下文或向后兼容的可选字段扩展，不得把赛事数据字段直接暴露为新的必填请求参数。

### 2.2 成功响应

以下示例来自 Demo 技术基线，只用于说明契约结构，不代表正式赛题的数据模型或问题范围。

```json
{
  "request_id": "req_xxx",
  "question": "查询客户C001的账户余额",
  "sql": "SELECT ... WHERE c.customer_id = :customer_id ...",
  "columns": ["customer_id", "account_balance"],
  "rows": [["C001", 6000000]],
  "summary": "客户C001当前有效账户余额合计为600.00万元。",
  "warnings": [],
  "error": null,
  "metadata": {
    "configured_mode": "hybrid",
    "executed_generator": "rule",
    "rule_matched": true,
    "route": "Rule",
    "failure_reason": null,
    "provider": null,
    "model": null,
    "llm_latency_ms": null,
    "semantic": null,
    "fallback": {
      "used": false,
      "reason": null,
      "fallback_generator": null
    }
  }
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `request_id` | string | 服务端生成的请求标识 |
| `question` | string | 保留用户原始问题 |
| `sql` | string/null | 已生成 SQL；生成前失败时为 null |
| `columns` | string[] | 结果列名 |
| `rows` | array[] | 按列顺序排列的 JSON 标量二维数组 |
| `summary` | string/null | 模板化中文业务结论 |
| `warnings` | string[] | 截断或非阻断提示 |
| `error` | object/null | 结构化错误 |
| `metadata` | object/null | 可选可解释信息，旧客户端可忽略 |

### 2.3 Metadata

Metadata 的当前字段：

| 字段 | 类型 | 含义 |
|---|---|---|
| `configured_mode` | string | 配置的 `rule`、`llm` 或 `hybrid` |
| `executed_generator` | string | 实际执行的 `rule` 或 `llm` |
| `rule_matched` | boolean | 是否命中经过验证的 Rule |
| `route` | string/null | 当前查询路径：`Rule` 或 `LLM` |
| `failure_reason` | string/null | 结构化失败分类 |
| `provider` | string/null | 当前仅支持 `deepseek` |
| `model` | string/null | 实际模型名 |
| `llm_latency_ms` | number/null | 两阶段模型调用总耗时 |
| `semantic` | object/null | LLM 解析出的业务语义 |
| `fallback` | object | 向后兼容字段 |

`semantic` 包含：

```json
{
  "intent": "monthly_transaction_summary",
  "business_domain": "transaction",
  "metrics": [
    "transaction_count",
    "transaction_inflow",
    "transaction_outflow",
    "net_transaction_flow"
  ],
  "dimensions": ["customer_id"],
  "filters": {"customer_id": "C001"},
  "time_range": {
    "start": "2026-06-01",
    "end": "2026-07-01"
  },
  "confidence": 0.97
}
```

`fallback` 只为旧客户端保持字段兼容。当前 Hybrid 不执行“LLM失败后回退Rule”，因此新流程中 `fallback.used` 保持 false；路由判断应读取 `rule_matched`、`route` 和 `failure_reason`。

### 2.4 错误响应

所有已知错误仍使用与成功响应相同的顶层结构：

```json
{
  "request_id": "req_xxx",
  "question": "帮我分析贷款",
  "sql": null,
  "columns": [],
  "rows": [],
  "summary": null,
  "warnings": [],
  "error": {
    "code": "CLARIFICATION_REQUIRED",
    "message": "请补充查询条件。",
    "retryable": false
  },
  "metadata": {
    "configured_mode": "hybrid",
    "executed_generator": "llm",
    "rule_matched": false,
    "route": "LLM",
    "failure_reason": "missing_parameter"
  }
}
```

当前状态码映射：

| HTTP | 错误代码 |
|---:|---|
| 400 | `INVALID_QUESTION`、`UNSUPPORTED_QUESTION`、`CLARIFICATION_REQUIRED`、`UNSUPPORTED_METRIC` |
| 403 | `SQL_REJECTED`、`ACCESS_DENIED` |
| 422 | `REQUEST_VALIDATION_ERROR` |
| 502 | `LLM_PROVIDER_ERROR` |
| 503 | `DATABASE_UNAVAILABLE`、`LLM_UNAVAILABLE` |
| 504 | `QUERY_TIMEOUT`、`LLM_TIMEOUT` |
| 500 | `INVALID_SEMANTIC_OUTPUT`、`INVALID_SQL_OUTPUT`、`QUERY_EXECUTION_ERROR`、`CONFIGURATION_ERROR`、`INTERNAL_ERROR` 等其余错误 |

API 不返回 Python Traceback、SQLite 原始错误、模型原始响应或密钥。

### 2.5 正式数据迁移兼容规则

五张官方表接入时必须遵守：

- `/api/v1/query` 路径和现有请求字段保持兼容；
- 正式数据库选择、机构范围和数据环境由后端配置与用户上下文决定；
- `columns` 和 `rows` 可以随问题变化，但必须保持 JSON 标量二维数组结构；
- 官方字段名不得无审查地直接成为用户展示名，展示层优先使用业务别名；
- 题库标准答案、分区标签和评测元数据不得进入普通查询响应；
- 新的解释、图表或审计信息优先放入可选 Metadata 或新增可选字段；
- 破坏性契约修改必须单独版本化，并经过前端、测试和项目负责人审核。

### 2.6 兼容别名

`POST /api/v1/ask` 仍映射到同一处理函数和同一 DTO，但不出现在 OpenAPI 中。新前端、测试说明和外部集成统一使用 `/api/v1/query`。

## 三、内部应用模型

内部模型定义在 `backend/app/application/models.py`，不依赖 FastAPI、Pydantic、SQLite、SQLGlot 或模型 SDK。

| 模型 | 主要用途 |
|---|---|
| `QueryCommand` | Pipeline 输入，包含问题、用户、会话和请求标识 |
| `QueryContext` | Resolver 输出，包含 Schema/Metric 文本和访问范围 |
| `GeneratedSQL` | Generator 输出，包含参数化 SQL、参数、警告和 Metadata |
| `UserContext` | Safety 输入的用户及表字段边界 |
| `SafetyResult` | Safety 检查结果 |
| `QueryResult` | Executor 返回的列、二维数据、行数、截断和耗时 |
| `FormattedResult` | 模板业务结论与展示警告 |
| `QueryOutcome` | Pipeline 统一输出 |
| `LLMRequest` / `LLMResponse` | 模型 Provider 的纯数据契约 |
| `BusinessSemantic` | 两阶段 LLM 的结构化业务语义 |
| `AuditEvent` | 请求开始、成功、拒绝或失败事件 |

## 四、Ports 契约

### 4.1 ContextResolver

```python
class ContextResolver(Protocol):
    def resolve(self, question: str) -> QueryContext: ...
```

当前 Demo 实现从 `config/schema.yml` 和 `config/metrics.yml` 选择相关上下文。正式阶段可以替换 Resolver 和配置来源，但不得让 Pipeline 直接读取官方 Excel。首版使用关键词映射，不是 RAG 或向量检索。

### 4.2 SQLGenerator

```python
class SQLGenerator(Protocol):
    def generate(self, question: str, context: QueryContext) -> GeneratedSQL: ...
```

`RuleSQLGenerator`、`LLMSQLGenerator` 和 `HybridSQLGenerator` 实现同一 Port。Hybrid 内部先 Rule，未命中才进入 LLM；LLM 失败是终止结果。

### 4.3 LLMProvider

```python
class LLMProvider(Protocol):
    def complete(self, request: LLMRequest) -> LLMResponse: ...
```

Provider 只负责发送模型请求并返回文本、模型名和耗时，不访问数据库、不做 SQL Safety，也不直接返回 SQL 领域对象。

### 4.4 SQLSafetyChecker

```python
class SQLSafetyChecker(Protocol):
    def validate(self, sql: str, user_context: UserContext) -> SafetyResult: ...
```

当前 Adapter 复用 SQLGlot AST 校验，只允许一条 SELECT/CTE/UNION 查询，拒绝写操作、系统表、未授权表和敏感字段。Safety 未通过时 Pipeline 不调用数据库。

### 4.5 DatabaseExecutor

```python
class DatabaseExecutor(Protocol):
    def execute_query(
        self,
        sql: str,
        parameters: dict[str, JsonScalar],
        max_rows: int = 1000,
    ) -> QueryResult: ...
```

当前 Demo SQLite Adapter 使用只读 URI、参数绑定、`query_only`、2秒查询期限和最多1000行结果，并统一转换数据库异常和 JSON 标量。正式数据库通过同一 Port 接入，前端和 Pipeline 不感知具体数据库文件。

### 4.6 ResultFormatter

```python
class ResultFormatter(Protocol):
    def format(self, question: str, result: QueryResult) -> FormattedResult: ...
```

当前实现只根据已知结果列生成确定性中文摘要，不调用 LLM。

### 4.7 AuditLogger

```python
class AuditLogger(Protocol):
    def record(self, event: AuditEvent) -> None: ...
```

当前使用 NoOp Adapter，调用位置已经固定，但尚未持久化。

## 五、组装与替换规则

生产 Adapter 只在 `backend/app/bootstrap/container.py` 组装。API 路由不创建数据库连接、Provider、Generator 或 Safety 实例。测试通过依赖覆盖和 Fake Adapter 替换具体实现。

任何 Port 参数、HTTP 顶层字段或错误代码的破坏性修改，都必须先更新本文和相关契约测试。新增可选 Metadata 字段应保持旧客户端可忽略。
