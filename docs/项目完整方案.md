# 言出数行——银行智能问数与协同分析系统项目方案

> 状态：正式赛题阶段方案。当前代码仍保留 Demo 技术基线，正式数据能力以 Issue #4 至 #6 的验收结果为准。

## 项目定位

言出数行是面向银行经营分析场景的智能问数与协同分析系统。银行业务人员用自然语言描述机构、指标、时间和比较需求，系统完成业务语义识别、指标口径匹配、SQL 生成、安全检查、数据库查询、结果解释和评测留痕。

项目不是单纯的 NL2SQL 工具。核心闭环是：

```text
业务问题
  -> 指标与 Schema 语义
  -> Rule First / LLM Extension
  -> SQL Safety
  -> 只读查询
  -> 业务结论与可视化
  -> Metadata 与评测记录
```

## 用户与场景

目标用户包括分支机构经营管理人员、业务分析人员、客户经理主管、风险和运营人员。他们能够提出经营问题，但不一定了解数据库结构、指标口径或 SQL。

系统重点解决：

1. 临时取数依赖数据团队，响应周期长；
2. 指标和衍生维度口径分散；
3. 固定 BI 无法覆盖灵活比较和追问；
4. 大模型可能产生字段幻觉、错误计算和危险 SQL；
5. 查询结果缺少过程解释和固定题库验证。

## 正式数据模型

正式阶段固定围绕五张官方表：机构信息表、指标清单表、衍生维度说明、指标数据表和问题答案清单。

| 数据部分 | 业务职责 | 系统用途 |
|---|---|---|
| 机构信息表 | 定义机构范围和层级 | 机构筛选、比较、排名 |
| 指标清单表 | 定义指标名称和基础口径 | 指标语义层和展示说明 |
| 衍生维度说明 | 定义比较和衍生规则 | 时间比较、派生计算和解释 |
| 指标数据表 | 保存机构、指标、日期和值 | 正式查询事实表 |
| 问题答案清单 | 保存官方问题及评测结果 | Gold SQL、回归和阶段验收 |

问题答案清单与业务查询数据库隔离。正式字段、键、数据类型和索引必须由数据审计确定，详见[数据库与指标字典](数据库与指标字典.md)。

## 系统架构

后端采用轻量 Ports & Adapters。每项能力可以独立替换和测试，Pipeline 只负责编排。

```text
Frontend
  -> FastAPI API
  -> QueryPipeline
       -> ContextResolver
       -> SQLGenerator
            -> RuleSQLGenerator
            -> LLMSQLGenerator -> LLMProvider
            -> HybridSQLGenerator
       -> SQLSafetyChecker
       -> DatabaseExecutor
       -> ResultFormatter
       -> AuditLogger
  -> QueryResponse
```

### Pipeline 职责

Pipeline 接收内部命令，按顺序调用接口，处理统一应用异常并返回结果。它不读取 Excel/YAML、不包含固定问题分支、不调用模型 SDK、不打开数据库连接、不解析 SQL AST，也不决定 HTTP 状态码。

### 查询路由

默认 Hybrid 采用 Rule First：高频、标准、口径确定的问题先使用已验证 Rule；未命中时才进入 LLM 语义解析和 SQL 生成。两条路径统一经过 SQL Safety 和只读 Executor。LLM 失败时返回结构化错误，不反向尝试 Rule。

## 安全与治理

- SQLGlot 解析并限制为单条只读查询；
- 限制允许表和敏感字段，Safety 失败时不访问数据库；
- Executor 使用参数绑定、只读连接、超时和最大返回行数；
- API 隐藏模型原始响应、数据库异常和堆栈；
- 密钥只保存在本地 `.env`；
- 官方数据、答案、详细映射和敏感结果不进入公开仓库；
- 前端只通过 API 取数，不读取原始 Excel 或直接连接数据库。

后续权限和审计应从基础用户角色、机构范围、敏感字段脱敏和查询留痕开始，不提前建设复杂权限平台。

## 可解释性与业务输出

响应 Metadata 用于展示配置模式、实际执行器、Rule 命中、查询路径、模型、业务意图、指标、过滤条件、时间范围、置信度、耗时和失败原因。

业务解释必须来源于查询结果和确定性规则。LLM 可以参与语义理解和 SQL 生成，但不得脱离数据自由生成经营结论。图表类型应由结果结构和业务问题共同决定，并显示指标单位和时间口径。

## 评测体系

固定问题集不是附属测试数据，而是版本基准。评测覆盖：

- 语义识别；
- Rule/LLM 路由；
- SQL 生成和安全校验；
- SQL 执行；
- 答案正确性；
- 响应时间；
- 难度和问题模式表现；
- 失败原因与版本差异。

训练分区用于开发与内部回归，测试分区用于阶段验收；工作簿实际包含的验证分区单独管理，不参与 Rule 或 Prompt 硬编码。详细规则见[评测规范](评测规范.md)。

## 产品形态

比赛页面应围绕“提问—理解—查询—结论—解释”组织，而不是展示开发工具：

1. 机构、指标和时间范围提示；
2. 自然语言输入与基于官方能力的推荐问题；
3. 业务结论和关键指标；
4. 查询结果与适用图表；
5. 默认折叠的 SQL 和技术详情；
6. 条件不足、无数据、模型失败、Safety 拒绝和服务异常状态。

当前 Streamlit 页面仍是技术基线，正式业务模块、指标卡和推荐问题必须等待官方数据字典与问题分类确认。

## Demo 技术基线

`demo-baseline` Release 保存了十张模拟业务表、三条 Demo 问题和完整技术链路。当前主线暂时保留 `data/processed/bankinsight.db`、`sql/schema.sql`、`config/schema.yml` 和 `config/metrics.yml`，供前端、CI 和迁移回归使用。

这些资源不代表正式赛题数据和指标范围。只有在正式数据库、后端切换、前端联调和回归评测全部通过后，才通过后续独立任务替换或删除。

## 团队协作

六项工作流为项目统筹、产品方案、前端集成、官方数据治理、题库语义与异常审核、固定题库评测。数据、语义和评测契约先行，产品与前端随后迁移，项目负责人维护公共接口和稳定主干。Gold SQL和路由实现由项目负责人另行指定技术成员负责，05号只维护业务语义资产。

核心 `application/`、`ports/`、`bootstrap/container.py`、数据库结构、公共 API 和配置变更必须经过项目负责人审核。任何成员不得绕过接口直接修改其他模块内部实现。

## 里程碑

1. 建立 Demo 基线和正式仓库规范；
2. 完成五张官方表审计、建库和双环境；
3. 完成题库表达映射、衍生口径和异常审核，再由技术成员建立Gold SQL；
4. 将正式数据接入现有查询主链路；
5. 完成产品迁移和比赛演示页面；
6. 建立固定题库回归并完成答辩版本收口。

详细负责人、依赖和验收标准见根目录 [task_plan.md](../task_plan.md)。

## 当前边界

当前没有把 RAG、向量数据库、LangGraph、多 Agent、复杂权限、桌面端、移动端或生产数据库迁移列为并行任务。只有当正式数据、Gold SQL 和评测基准稳定后，团队才根据赛题价值创建独立 Issue 评估新增能力。
