# BankInsight：银行业智能问数与指标洞察平台项目方案

## 一、项目定位

### 一句话定位

BankInsight 是一个面向银行经营分析场景的智能问数与指标洞察平台，让分行管理者、客户经理主管、运营和风控人员用自然语言完成从“提出问题”到“获得可信结论”的全过程。

### 目标用户与核心任务

| 用户 | 高频任务 | 系统提供的帮助 |
|---|---|---|
| 总分行经营管理者 | 看规模、增量、排名和结构变化 | 自动匹配指标口径，完成跨分行对比和趋势归因 |
| 客户经理主管 | 看客户分层、AUM、流失和经理绩效 | 关联客户、账户、产品与经理数据，定位重点客群 |
| 风控人员 | 看逾期、不良和风险事件 | 提供风险趋势、群体分布和下钻明细 |
| 渠道运营人员 | 看 App、网银、柜面等渠道表现 | 计算活跃、转化和迁移，识别渠道异常 |
| 数据分析人员 | 响应临时取数并维护指标口径 | 降低重复 SQL 工作，保留完整查询血缘与审计记录 |

### 产品边界

首版聚焦“经营分析与风险观察”，不承担授信审批、自动交易、客户触达等高风险决策。系统输出用于辅助分析，敏感客户明细默认脱敏，并通过角色权限控制访问范围。

## 二、要解决的问题与项目价值

### 传统取数的主要问题

1. 业务人员不会 SQL，临时需求需要排队，问题从提出到拿到结果可能跨越多个沟通轮次。
2. 银行表多、字段多、历史口径复杂，业务人员无法知道“客户数”是开户客户、有效客户还是月活客户。
3. 同一指标可能在不同部门存在不同过滤条件、统计时点和去重方式，导致会议中出现“同指标不同数”。
4. 固定 BI 看板适合监控已知问题，却难以响应“为什么下降”“哪些客群贡献最大”等临时下钻。
5. 大模型直接生成 SQL 会产生字段幻觉、错误 JOIN、重复计数、越权查询和不可解释结果。

### 系统价值

- **效率价值**：把标准经营问题从“提需求—排期—写 SQL—核数”缩短为分钟级自助分析。
- **治理价值**：指标、维度、过滤条件和版本集中维护，查询时明确展示采用的口径。
- **分析价值**：从返回数据升级为“趋势—异常—贡献—建议追问”的分析链路。
- **安全价值**：将只读权限、敏感字段、查询范围、成本预算和审计日志嵌入执行流程。
- **组织价值**：沉淀高频问题、标准 SQL 和业务定义，形成可复用的银行经营知识库。

## 四、系统整体架构

### 分层架构

```text
交互层        问数工作台 | 指标中心 | 查询历史 | 管理后台
                              |
智能编排层    问题理解 -> 查询规划 -> SQL 生成 -> 修复 -> 解释 -> 追问
                    |             |          |
语义治理层    指标目录       Schema 元数据   示例 SQL/业务术语
                              |
可信执行层    AST 校验 -> 权限/成本检查 -> 只读执行 -> 结果一致性校验
                              |
数据层        客户 | 账户 | 交易 | 贷款 | 理财 | 渠道 | 风险 | 组织
                              |
治理与观测    权限、脱敏、审计、模型调用日志、成功率、准确率、响应时延
```

### 模块输入输出

| 模块 | 作用 | 输入 | 输出 |
|---|---|---|---|
| 用户提问 | 接收问题、上下文和展示偏好 | 自然语言、角色、当前会话 | 标准请求对象 |
| 问题理解 | 抽取意图、指标、维度、时间、筛选、排序 | 原问题、会话历史、业务词典 | `QueryIntent` 结构化意图 |
| 指标体系 | 管理定义、公式、粒度、默认过滤和版本 | 指标名称或别名 | 指标 ID、SQL 表达式、口径说明 |
| Schema 管理 | 管理表字段、中文别名、关系、敏感级别 | 数据库元数据、人工标注 | 可检索 Schema 目录 |
| 表字段召回 | 找到与问题最相关的表、字段和连接路径 | 结构化意图、指标依赖 | Top-K 表字段、Join Path、召回分数 |
| NL2SQL | 基于受限上下文生成查询 | 问题、意图、指标定义、候选 Schema、示例 | SQL、参数、生成依据 |
| SQL 安全校验 | 防止写操作、越权、全表扫描和敏感字段泄露 | SQL、用户权限、查询预算 | 通过/拒绝、风险原因、改写建议 |
| 执行与修复 | 在只读连接执行，依据真实报错有限次修复 | 已通过 SQL、数据库错误 | 结果集、执行统计、修复记录 |
| 结果一致性 | 校验空结果、重复行、单位、分母和总分关系 | 结果、指标元数据、查询计划 | 质量标签、告警、可信度 |
| 结果解释 | 生成证据约束的业务摘要 | 聚合结果、对比值、指标定义 | 结论、证据、风险提示 |
| 可视化 | 根据数据结构和分析意图选图 | 结果字段类型、维度数、意图 | 图表类型与 ECharts 配置 |
| 推荐追问 | 根据当前结果生成下一步可执行问题 | 当前问题、异常、可用维度 | 3 个追问及推荐理由 |
| 前端展示 | 统一呈现问数全过程 | 上述全部输出 | 工作台、指标卡、图表、SQL、解释 |

### 关键连接关系

问题理解先连接指标体系，把“贷款逾期率”变成受治理的指标；再连接 Schema 召回，限定可使用的表和字段。SQL 生成不能直接访问全量 Schema，只能依据召回结果和批准的连接路径。SQL 通过安全校验后才可执行。解释模块读取真实结果和指标定义，不能读取模型臆测出的中间结论。推荐追问重新进入同一问数链路，因此每个追问都应可以实际执行。

## 五、银行业务指标体系

完整字段和公式见[数据库与指标字典](数据库与指标字典.md)。首版七个主题如下。

| 主题 | 典型业务问题 | 核心指标 | 主要数据表 | 示例问法 |
|---|---|---|---|---|
| 客户分析 | 客户规模、分层、活跃和流失 | 有效客户数、新增客户数、月活客户数、AUM、流失率 | `customer_info`、`account_info`、`transaction_detail` | “本季度华东地区高净值客户增长主要来自哪些分行？” |
| 存款分析 | 余额、增量、结构和集中度 | 存款余额、日均存款、新增存款、活期占比、客户集中度 | `account_info`、`transaction_detail`、`branch_info` | “近 6 个月各分行存款余额和环比变化如何？” |
| 贷款分析 | 规模、投放、逾期和不良 | 贷款余额、新增贷款、逾期金额、逾期率、不良率 | `loan_info`、`customer_info`、`branch_info` | “近 3 个月贷款逾期率最高的客户群体是什么？” |
| 理财产品 | 销量、持有、到期和客户转化 | 申购金额、持有金额、购买客户数、复购率、产品到期金额 | `wealth_product`、`product_purchase` | “本月各风险等级理财产品销量及客户转化情况如何？” |
| 分行经营 | 规模、增量、排名和目标达成 | 客户数、存贷款余额、AUM、收入代理指标、综合得分 | `branch_info` 与各事实表 | “本季度经营指标排名提升最快的 5 家分行是谁？” |
| 渠道运营 | 活跃、转化、渠道迁移和失败 | DAU/MAU、交易用户数、转化率、渠道交易额、失败率 | `channel_behavior`、`transaction_detail` | “App 月活下降集中在哪些地区和客户层级？” |
| 风险预警 | 逾期、异常交易、风险事件和处置 | 风险客户数、高风险事件数、逾期率、事件关闭率 | `risk_event`、`loan_info`、`transaction_detail` | “近 30 天高风险事件上升主要由哪类事件贡献？” |

口径设计遵循五要素：统计对象、时间口径、聚合方式、默认过滤、数据粒度。每个结果都展示这五项和指标版本。

## 六、模拟数据库设计

首版采用 10 张表，既能覆盖经营分析，又控制关联复杂度。

| 表 | 粒度 | 主键 | 主要外键 | 支持的问题 |
|---|---|---|---|---|
| `branch_info` | 一行一分行 | `branch_id` | `parent_branch_id` | 地区、分行层级、分行排名 |
| `customer_manager` | 一行一客户经理 | `manager_id` | `branch_id` | 经理绩效、管户规模和客户归属 |
| `customer_info` | 一行一客户 | `customer_id` | `branch_id`,`manager_id` | 客群、等级、地区、开户和流失 |
| `account_info` | 一行一账户 | `account_id` | `customer_id`,`branch_id` | 存款余额、账户类型和状态 |
| `transaction_detail` | 一行一交易 | `transaction_id` | `account_id`,`customer_id`,`branch_id` | 交易趋势、收支、渠道和异常 |
| `loan_info` | 一行一贷款合同 | `loan_id` | `customer_id`,`branch_id`,`manager_id` | 贷款余额、逾期、五级分类 |
| `wealth_product` | 一行一理财产品 | `product_id` | - | 产品期限、风险等级和状态 |
| `product_purchase` | 一行一申购记录 | `purchase_id` | `product_id`,`customer_id`,`branch_id` | 理财销量、持有、复购和到期 |
| `channel_behavior` | 一行一客户一次渠道事件 | `behavior_id` | `customer_id`,`branch_id` | 登录、访问、交易、渠道转化 |
| `risk_event` | 一行一风险事件 | `risk_event_id` | `customer_id`,`loan_id`,`branch_id` | 风险类型、等级、状态和处置 |

关系主线为“分行—客户经理—客户—账户—交易”；贷款、理财购买、渠道行为、风险事件围绕客户连接。所有事实表保留 `branch_id`，用于重现业务发生时归属，避免客户后续转移分行导致历史统计变化。

模拟数据不会复制真实客户数据。生成器应满足余额非负、合同日期有序、风险等级与事件概率相关等约束，并预埋三个可解释事件：某分行逾期率连续上升、某地区 App 活跃突降、某类理财产品到期后复购不足。

### 推荐追问示例

对问题 6 可推荐：“连续两个月上升的是哪些分行？”“这些分行的逾期主要集中在哪些客群？”“逾期上升是否伴随新增贷款结构变化？”推荐项必须附带已解析的指标和维度，点击后重新执行，而不是只生成一句文本。

### 单个场景的系统输出模板

1. **答案摘要**：一句话回答问题，包含核心数值和时间范围。
2. **图表与数据**：默认展示最适图表，可切换表格并下载脱敏结果。
3. **业务洞察**：描述趋势、异常和主要贡献，列出证据数值。
4. **口径卡片**：指标公式、统计粒度、过滤条件、更新时间和版本。
5. **SQL 与血缘**：展示生成 SQL、使用表、连接路径、执行耗时和修复次数。
6. **推荐追问**：提供三条可执行的下一步分析。

## 八、NL2SQL 技术路线

### 可信问数流水线

1. **问题规范化**：解析“最近”“本季度”“我行”等相对表达，结合当前日期、用户机构和会话上下文形成明确问题。必要性在于自然语言中大量条件被省略。
2. **意图抽取**：输出指标、维度、时间、过滤、排序、粒度和分析类型。使用 Pydantic/JSON Schema 约束模型输出，避免自由文本难以校验。
3. **指标口径匹配**：把“逾期率”映射到指标 ID、分子分母、统计时点和默认过滤。若存在多个版本，先展示消歧选项或按用户部门默认版本处理。
4. **Schema Linking**：采用关键词/BM25 与向量相似度混合召回字段，再根据外键图补齐最短合法 Join Path。模型只看到 Top-K 候选，降低字段幻觉。
5. **查询计划生成**：SQL 前先输出事实表、维度表、粒度、聚合顺序和 Join Cardinality。对同比、漏斗、先客户后经理聚合等问题尤其重要。
6. **Prompt 组装**：输入原问题、结构化意图、指标定义、候选字段、允许的 Join、SQL 方言、权限、相似示例和输出格式，不输入无关全库 Schema。
7. **SQL 生成**：模型返回参数化 SQL、字段解释和引用依据。温度设低，禁止创造未提供的表字段。
8. **语法与语义检查**：用 SQLGlot 解析 AST，验证字段存在、别名可解析、聚合与 Group By 合法、Join 条件完整，必要时用数据库 `EXPLAIN`。
9. **安全与成本检查**：仅允许单条 SELECT/CTE；禁止 DDL/DML、系统表、未授权敏感列；强制行数上限、超时、扫描量预算和只读连接。
10. **执行与有限修复**：执行失败后，只把脱敏的数据库错误、原 SQL 和允许 Schema 交给修复器；最多两次。每次修改记录 diff，安全校验重新执行。
11. **结果一致性校验**：检查空结果、分母为零、负余额、重复主键、总分不一致、单位异常和时间覆盖不足。高风险异常不自动下结论。
12. **解释与追问**：先由程序计算最大值、变化率、贡献度等结构化证据，再让模型组织业务语言；追问从可用指标和维度中生成并预校验可执行性。

### Prompt 模板核心结构

```text
角色：银行经营分析 SQL 规划器
任务：基于已批准语义层生成 {dialect} 查询
用户问题：{question}
结构化意图：{intent_json}
指标定义：{metric_definitions}
允许表字段：{retrieved_schema}
允许连接：{join_paths}
权限与限制：{access_policy}
相似示例：{few_shot_examples}
要求：只引用给定字段；使用参数；先按业务粒度聚合；只输出约定 JSON
```

### 降低幻觉的控制点

- 生成前：指标消歧、Schema 限定、Join Path 限定、示例检索。
- 生成中：结构化输出、低温度、SQL 方言固定、参数化时间条件。
- 生成后：AST、元数据、权限、`EXPLAIN` 和结果规则五重检查。
- 展示时：公开口径、SQL、表血缘、修复过程与可信度，不把模型文本当作事实来源。

## 九、项目创新点

| 创新点 | 具体解释 | 实现方式 |
|---|---|---|
| 银行指标语义层驱动 | 先确定业务口径再生成 SQL，解决“SQL 对但业务含义错” | YAML 指标目录维护公式、粒度、过滤、版本、依赖表和别名 |
| 混合 Schema Linking | 同时利用业务词匹配、向量语义和关系图，减少无关 Schema | BM25 + Embedding Top-K，再用外键图补齐最短 Join Path |
| 查询计划先行 | 把复杂问题拆成粒度、事实表和聚合步骤，减少重复计数 | 生成结构化 `QueryPlan`，规则引擎检查 Join Cardinality |
| 可审计安全网关 | 安全不是一句“仅 SELECT”，而是贯穿权限、成本和脱敏 | AST 白名单、RBAC、敏感列策略、限时限行、审计日志 |
| 证据约束的业务解释 | 解释中的每个数字可追溯到结果字段，降低语言模型编造 | 程序生成证据 JSON，模型只能引用 evidence ID |
| 自动修复但不静默 | 查询报错可修复，同时展示修改原因和次数 | 数据库错误分类、最多两次修复、每次重新过安全检查 |
| 可执行推荐追问 | 推荐问题不是装饰文本，而是已通过指标和 Schema 预检查 | 基于当前异常和可用下钻维度生成候选，预跑意图解析 |
| Agentic 分析闭环 | 多代理/多步骤各自承担可验证职责，而非一个模型包办 | Planner、Retriever、SQL Generator、Critic、Analyst 状态机 |

## 十、最终展示形态

### 前端信息架构

- 左侧窄栏：会话历史、主题入口、收藏问题。
- 中间主工作区：提问框、答案摘要、图表与结果表格，支持继续追问。
- 右侧可折叠证据栏：指标口径、SQL、表血缘、执行状态、可信度。
- 顶部：当前机构、数据日期、角色权限和环境标识。
- 管理页：指标中心、Schema 中心、示例问法、查询审计和模型评估。

提问区提供业务主题提示，但不做营销式首页；系统打开即进入可操作工作台。SQL 默认折叠，评委可展开查看。图表自动推荐但允许切换，表格支持排序、筛选和脱敏下载。

### 三个答辩演示案例

**案例 A：分行逾期率异常归因**

提问“近 3 个月各分行贷款逾期率如何变化？”系统展示趋势并标记某分行连续上升；点击追问“主要来自哪些客户群体？”下钻到职业、客户等级或贷款类型；最后展开口径和 SQL，证明逾期率按余额而非合同数计算。价值是同时展示趋势、异常、归因和可信口径。

**案例 B：App 活跃下降定位**

提问“近 8 周手机银行 MAU 和交易转化率是否下降？”系统识别活跃下降起点；追问“下降主要来自哪些地区和客户等级？”用瀑布图展示贡献。价值是表现多指标、时间对比、客户分层和运营决策支持。

**案例 C：理财到期承接机会**

提问“未来 30 天到期的理财金额按分行怎么分布？”系统展示到期金额 Top 分行和客户数；追问“其中风险等级匹配且近一年有复购的客户有多少？”系统在权限允许时返回脱敏客户群规模。价值是把固定报表升级为可连续探索的经营机会分析。

### 答辩可展示的系统指标

- SQL 执行成功率、首轮成功率、平均修复次数。
- 指标识别准确率、表字段召回 Recall@K、执行结果准确率。
- 危险 SQL 拦截率、越权字段拦截率。
- 平均响应时延和 P95 时延。
- 推荐追问点击率可作为后续产品指标，竞赛阶段用可执行率替代。

## 十一、题目 14 与题目 22 的客观比较

| 维度 | 14：Agentic AI + 图谱穿透 + 财报反欺诈 | 22：银行智能问数 + NL2SQL |
|---|---|---|
| 核心价值 | 跨文档、实体和关系识别潜在欺诈线索 | 降低银行经营取数门槛，统一指标并支持临时分析 |
| 展示冲击力 | 图谱穿透和欺诈链路视觉效果强，研究感更突出 | 问题—SQL—图表—洞察闭环直观，业务人员容易理解 |
| 数据要求 | 财报、工商关系、股权、公告、处罚、司法等多源数据 | 可用结构化模拟银行数据，数据边界可控 |
| 关键难点 | 实体消歧、图谱构建、欺诈标签、证据链和因果误判 | 指标口径、Schema Linking、SQL 正确性和安全治理 |
| 验证难度 | “发现关系”不等于“证明欺诈”，缺少高质量真值标签 | 可用标准问题—SQL—结果集构建明确评测集 |
| Demo 完整度 | 容易做到图谱展示，难做到反欺诈结论可信闭环 | 四周内较容易形成稳定、可重复的完整产品演示 |
| 简历适配 | 更适合算法、知识图谱、风控研究岗位 | 更适合数据分析、数据产品、AI 产品和数据开发岗位 |

### 对题目 14 的判断

优势在于议题新、视觉冲击强，能够体现多 Agent、知识图谱、文档解析和证据检索能力。难点是数据链路很长：公开财报只能反映部分信息，工商股权和司法数据存在可得性与授权问题，实体名称需要消歧，真实欺诈标签稀缺。项目最容易遇到的风险是把“关联异常”表述成“欺诈事实”，评委会追问验证标准、误报率和证据合法性。

### 对题目 22 的判断

22 的优势是需求真实、数据结构清楚、评价指标可量化、产品闭环完整。即使使用模拟数据，也可以对每个问题建立标准 SQL 和预期结果，客观计算执行准确率；SQL 安全、指标口径和交互链路也都有可见成果。因此它更适合四周内做成稳定 Demo，也更容易在数据分析/产品面试中讲清“业务问题—设计决策—量化验证—产品价值”。

### 吸收题目 14 的 Agentic AI 思路

选 22 不等于放弃 Agentic AI。可以把单模型调用升级为有状态编排：Intent Agent 解析问题，Metric Agent 匹配口径，Schema Agent 召回字段，SQL Agent 生成查询，Critic Agent 校验计划，Analyst Agent 解释结果。各 Agent 必须交换结构化对象、设置最大步数并保留日志。这样吸收了 14 的多步骤推理，但仍有清晰、可验证的落地目标。

### 选择建议

如果团队目标是算法论文感、已有可靠多源数据和图谱经验，14 更有挑战性；如果目标是竞赛中交付完整系统，并兼顾数据分析/产品实习作品集，22 的完成度和可解释性更有优势。基于当前目标，建议主选 22，并把 Agentic 编排作为技术增强点。

## 十三、简历包装方向

### 数据分析实习版本

- 设计银行经营分析智能问数项目，围绕客户、存贷款、理财、渠道和风险搭建 10 张主题表与 20+ 项指标语义层，统一指标公式、统计粒度和默认过滤条件。
- 构建覆盖聚合、排名、同比环比、多表关联和异常归因的 NL2SQL 评测集，引入 Schema Linking、Join Path 约束与结果一致性校验，对比直接 Prompt 基线并跟踪执行准确率与字段召回率。
- 完成分行逾期率归因、手机银行活跃下降定位和理财到期承接 3 个端到端案例，自动输出图表、证据化业务结论和可执行追问，形成可交互经营分析工作台。

### 产品 / AI 产品实习版本

- 从银行临时取数效率低和指标口径不一致出发，设计“提问—理解—取数—校验—解释—追问”产品闭环，明确经营管理者、客户经理主管、运营和风控四类角色的核心任务。
- 规划指标中心、Schema 管理、问数工作台和查询审计四个产品模块，设计 SQL 可解释侧栏、指标口径卡、错误修复记录、敏感字段脱敏与只读权限策略。
- 推动四周 MVP 路线与评测方案，将 SQL 成功率、执行准确率、危险查询拦截率、响应时延和追问可执行率作为核心指标，完成 3 条答辩演示链路和产品原型。

简历中的数字应在项目完成后替换为真实测量结果，不提前写“准确率提升 30%”等无法验证的数字。

## 首版验收标准

1. 15 个标准问题均有结构化意图、Gold SQL、期望图表与解释要点。
2. 至少 20 个指标有唯一 ID、公式、粒度、默认过滤、依赖字段和版本。
3. SQL 安全测试覆盖 DML/DDL、注释绕过、多语句、敏感列和超限查询。
4. 三个演示案例可重复运行，结果由预埋业务事件支撑。
5. 评测报告同时展示正确性、安全性、时延和基线对比。
6. 任何自然语言结论中的数值都能追溯到结果字段或派生证据。
