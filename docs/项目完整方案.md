# BankInsight 当前项目方案

> 产品版本：0.5.2
> 校准日期：2026-07-14
> 状态：当前正式方案，以运行代码、测试和接口契约为准

## 一、项目定位

BankInsight 是面向银行经营分析场景的智能问数与指标洞察平台。它帮助不熟悉 SQL 和底层表结构的业务人员，通过自然语言获得可执行 SQL、真实查询结果、模板化业务结论和可解释的查询路径。

项目不是单纯的“自然语言生成 SQL”工具。当前闭环包括：

```text
业务问题
  -> Schema 与指标上下文
  -> Rule First / LLM Extension
  -> SQLGlot 安全检查
  -> 只读 SQLite 查询
  -> 结果表格与模板结论
  -> Metadata 技术详情
```

## 二、目标用户与问题

### 目标用户

- 分行经营管理人员；
- 客户经理主管；
- 数据运营与经营分析人员；
- 风险管理人员；
- 需要临时取数但不会编写 SQL 的银行业务人员。

### 需要解决的问题

1. 业务人员依赖数据团队取数，临时需求响应慢。
2. 表、字段和指标口径分散，同名指标容易产生不同理解。
3. 固定 BI 看板无法覆盖临时追问。
4. 大模型直接生成 SQL 可能产生字段幻觉、错误 JOIN 和危险语句。
5. 只返回 SQL 或表格不能说明模型做了什么，也不便于业务复核。

### 当前产品价值

- 通过指标和 Schema 配置统一业务语义上下文；
- 用 Rule 保证高频标准查询稳定、低成本和口径确定；
- 用 LLM 扩展 Rule 尚未覆盖的自然语言；
- 用统一 Safety 和只读 Executor 阻止危险查询；
- 用 Metadata 展示路由、模型、语义、置信度、耗时和失败原因；
- 用模板 Formatter 输出可复核的确定性结论。

## 三、当前架构

后端采用轻量 Ports & Adapters，不使用微服务、完整 DDD、消息队列、LangGraph 或多 Agent。

```text
Streamlit Frontend
        |
        v
FastAPI /api/v1/query
        |
        v
QueryPipeline
  |-- ContextResolver -> YAML Schema / Metrics
  |-- SQLGenerator
  |     |-- RuleSQLGenerator
  |     |-- LLMSQLGenerator -> DeepSeek LLMProvider
  |     `-- HybridSQLGenerator（Rule First）
  |-- SQLSafetyChecker -> SQLGlot AST
  |-- DatabaseExecutor -> Readonly SQLite
  |-- ResultFormatter -> Template Summary
  `-- AuditLogger -> NoOp Adapter
```

### 分层职责

| 层 | 当前职责 |
|---|---|
| `api/` | HTTP DTO、路由、状态码和统一错误响应 |
| `application/` | 框架无关模型、统一异常和 Pipeline 编排 |
| `ports/` | Resolver、Generator、LLM、Safety、Database、Formatter、Audit 接口 |
| `adapters/` | YAML、Rule/LLM/Hybrid、DeepSeek、SQLGlot、SQLite、模板和 NoOp 实现 |
| `bootstrap/` | 唯一生产组装入口 |
| `frontend/` | 只通过 HTTP API 查询的 Streamlit 产品页面 |

Pipeline 不实现具体规则、不读取配置、不调用模型 SDK、不操作数据库、不解析 AST。

## 四、查询模式

### Rule 模式

只允许三个经过验证的标准问题。Rule 生成参数化 SQL，适合高频、固定口径场景，不需要模型密钥。

### LLM 模式

所有问题直接进入 DeepSeek。第一阶段将问题解析为结构化业务语义，第二阶段基于语义、指标和 Schema 生成参数化 SQLite SQL。该模式主要用于模型能力验证。

### Hybrid 模式

默认采用 Rule First、LLM Extension：

```text
命中 Rule -> Rule SQL -> Safety -> Database
未命中 Rule -> LLM Semantic -> LLM SQL -> Safety -> Database
LLM 失败 -> 结构化错误，流程结束
```

LLM 失败后不会再尝试 Rule。Hybrid 的价值是确定性能力与智能扩展能力分工，而不是两套生成器相互兜底。

## 五、数据与指标

### 数据模型

当前 SQLite DDL 包含10张表：分行、客户经理、客户、账户、交易、贷款、理财产品、理财购买、渠道行为和风险事件。

演示数据库当前只填充支撑三条 Gold 问题的小规模确定性数据：3名虚构客户、3家分行、1名客户经理、4个账户和4笔交易。贷款、理财、渠道与风险表目前为空，因此页面中的相关业务模块属于产品框架，不代表完整查询能力已经开放。

### 指标语义层

`config/metrics.yml` 已登记19项指标，覆盖客户、存款、交易、贷款、理财、渠道和风险主题。当前完成端到端验证的范围集中在：

- 有效客户数；
- 客户有效账户余额；
- 成功交易笔数；
- 交易流入金额；
- 交易流出金额；
- 交易净流入。

其余指标是已登记口径，不应描述为已经稳定支持。详细字段、公式、数据状态和规划项见[数据库与指标字典](数据库与指标字典.md)。

## 六、可靠性与安全

### SQL Safety

- 使用 SQLGlot 解析 AST；
- 只允许单条 SELECT、CTE 或 UNION 查询；
- 拒绝 INSERT、UPDATE、DELETE、DROP、CREATE、ALTER、MERGE 和命令语句；
- 检查允许表、系统表和敏感字段；
- Safety 未通过时不调用数据库。

### 数据库执行

- SQLite URI 只读连接；
- 开启 `PRAGMA query_only`；
- 使用命名参数绑定动态值；
- 查询期限2秒；
- 最大返回1000行；
- 自动关闭连接并统一转换错误；
- 数据库值转换为 JSON 标量。

### 密钥与数据边界

- DeepSeek Key 只从本地 `.env` 读取；
- `.env`、虚拟环境、日志和本地归档不进入 Git；
- 仓库只允许提交模拟数据；
- API 不返回第三方原始错误、模型原始响应或 Traceback。

## 七、可解释性

API 的可选 Metadata 用于回答“这次查询是谁完成的、模型理解了什么、为什么失败”：

- 配置模式和实际执行器；
- 是否命中 Rule；
- Rule 或 LLM 路由；
- 模型提供方与模型名；
- 业务意图、领域、指标、维度、过滤和时间范围；
- 模型置信度与耗时；
- 结构化失败原因。

旧 `fallback` 字段仅为客户端兼容保留，不代表新 Hybrid 仍有 LLM 到 Rule 的回退。

## 八、产品页面

Streamlit 单页包含：

1. BankInsight 品牌和银行经营分析定位；
2. 客户、存款、贷款、理财、风险和经营六类场景选择器；
3. 从 SQLite 实时读取的经营概览；
4. 自然语言查询输入与推荐问题；
5. 业务结论、关键指标、结果表格、生成 SQL；
6. 默认折叠的技术详情。

场景选择器中的未开放模块会明确提示后续开放，不会把产品设计当成已实现能力。前端不访问数据库、不生成 SQL，也不绕过 Pipeline。

## 九、当前演示案例

### 有效客户数量

展示 Rule 命中、客户口径、只读聚合查询和确定性结论。演示结果为2户。

### 客户账户余额

展示客户与账户关联、参数绑定、安全检查和金额单位转换。C001演示结果为600.00万元。

### 月度交易汇总

展示时间范围、交易四指标、条件聚合和交易净流。C001在2026年6月有3笔成功交易，流入10万元、流出5万元、净流入5万元。

### LLM 扩展

非标准同义问法可在 Hybrid 未命中 Rule 后进入 DeepSeek。其成功范围取决于当前指标、Schema、查询条件和模型输出，不承诺任意银行问题都能完成。

## 十、当前创新点

1. **指标语义优先**：模型只看到从配置选出的指标与 Schema 上下文。
2. **Rule First Hybrid**：确定性高频查询优先，模型只扩展未知问题。
3. **两阶段 LLM**：业务语义和 SQL 生成分离，便于校验与解释。
4. **统一安全主链路**：Rule 与 LLM 生成的 SQL 都经过相同 Safety 和只读执行器。
5. **过程可解释**：Metadata 展示路由、语义、耗时和失败原因。
6. **确定性业务解释**：结论由查询结果模板生成，不让模型自由编造。
7. **可替换工程接口**：Provider、Generator、Safety、Database 和 Formatter 可独立实现与测试。

## 十一、验证基线

当前基线为87项自动化测试，覆盖：

- 数据库初始化和约束；
- SQLite 只读、参数、限行和超时；
- SQL Safety 合法与危险查询；
- Rule、LLM 和 Hybrid 路由；
- LLM 输出校验和结构化失败；
- Pipeline 编排与架构依赖；
- FastAPI 请求、响应和错误契约；
- Streamlit 页面、状态恢复和稳定性保护。

真实 DeepSeek Smoke Test 是显式脚本，没有本地 Key 时显示 `SKIPPED`，不伪造调用成功。

## 十二、当前限制与后续方向

当前没有实现：

- RAG、向量数据库或复杂 Schema Linking；
- LangGraph、多 Agent 或任务状态机；
- 完整权限、行列级授权和持久化审计；
- PostgreSQL/MySQL 生产连接；
- 贷款、理财、渠道和风险主题的完整演示数据与 Gold 问题；
- Docker、正式桌面端或移动端 App；
- 自动生成图表和推荐追问的完整后端能力。

下一步不应盲目增加功能。应先从竞赛评委视角评审业务价值、演示证据、模型必要性和可量化效果，再决定补充数据主题、设计规范或部署能力。
