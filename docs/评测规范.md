# 言出数行评测规范

> 状态：正式评测基座规范。题目、标准答案和逐题结果属于受控资产，不进入公开仓库。

## 评测目标

固定题库用于衡量版本变化是否真正提升系统，而不是只证明少量演示问题能够运行。每次 Rule、Prompt、Schema、路由、数据库、Safety 或 Formatter 变化后，都应在同一基准上回归并输出可比较结果。

## 数据分区

官方工作簿实际包含200道问题，分为训练120、验证40、测试40。

| 分区 | 用途 | 开发可见范围 | 禁止事项 |
|---|---|---|---|
| 训练 | 语义审核、技术实现、开发调试和内部回归 | 业务、数据、测试负责人按职责和权限使用 | 不得把标准答案硬编码到产品 |
| 验证 | 阶段内参数与方案比较 | 受控评测流程使用 | 不得写入 Rule、Prompt 或推荐问题 |
| 测试 | 阶段验收和最终性能评估 | 验收时按权限使用 | 不得用于开发调参或答案泄漏 |

工作簿中的“问题类型”字段是数据事实，不应被改写成不存在的官方划分，也不能把验证或测试内容混入训练代码。

## 评测对象

每道题按以下链路记录结果：

```text
Question
  -> Semantic Parsing
  -> Route Selection
  -> SQL Generation
  -> SQL Safety
  -> Database Execution
  -> Answer Normalization
  -> Gold Comparison
```

### 必须记录的字段

- 题目编号和受控分区；
- 问题模式和难度；
- 目标机构、指标、时间、比较与筛选；
- 配置模式、实际路由和 Rule 命中；
- SQL 生成状态和 Safety 结果；
- 执行状态、耗时和返回行数；
- 答案比对结果；
- 失败阶段和标准化失败原因；
- 代码版本、配置版本、模型和数据库版本。

## 核心指标

| 指标 | 定义 |
|---|---|
| 语义识别正确率 | 机构、指标、时间和比较方式全部正确的题目占比 |
| Rule 命中准确率 | 应由 Rule 处理且被正确命中的题目占比；同时记录误命中 |
| SQL 生成成功率 | 生成语法有效、字段存在且参数完整 SQL 的题目占比 |
| Safety 通过率 | 合法问题生成结果通过安全层的占比；危险样例另算拦截率 |
| SQL 执行成功率 | 通过 Safety 后成功执行并返回结构化结果的题目占比 |
| 答案正确率 | 规范化结果与标准答案一致的题目占比 |
| 端到端成功率 | 从问题输入到正确答案完整成功的题目占比 |
| 响应时间 | 端到端 P50、P95、最大值，并区分 Rule 与 LLM |

所有指标都应按问题难度、问题模式、指标类型和路由拆分，避免总体平均数掩盖复杂题失败。

## 答案比对

答案比对不得只做原始字符串相等。Issue #5提供已确认的业务口径和异常信息，Issue #6据此实现并运行答案规范化与比对：

- 数值单位和允许误差；
- 百分比、小数位和舍入；
- 日期与时间范围格式；
- 排名、并列和排序方向；
- 表格列名、列顺序和行顺序；
- 空值、无数据与零值的区别；
- 文本答案中的机构和指标别名。

规范化规则必须版本化，并通过人工抽样验证，防止“宽松比对”虚增准确率。

## Gold SQL 管理

Gold SQL可以借助Codex生成初稿，但由项目负责人指定的技术成员负责生成和维护，数据负责人核对字段和关联，测试负责人核对预期结果。05号只确认其中引用的指标、时间、公式和单位是否符合已审核业务资产，不编写或维护SQL。每条Gold SQL应记录技术审核状态和适用数据库版本。

测试分区题目及其答案不得写入 Rule Repository、Prompt 示例、前端推荐问题或公开测试。公开仓库可以保存评测框架、空白 Schema 和脱敏汇总，不保存逐题标准答案。

## 回归门槛

每个影响查询链路的 Pull Request 至少完成：

1. 现有 Demo 自动化测试；
2. 受影响问题模式的训练回归；
3. SQL Safety 正常与危险样例；
4. Demo/Real 环境切换；
5. 与上一个发布版本的准确率、耗时和失败原因比较。

阶段发布前再运行受控验证和测试分区。出现端到端正确率下降、危险 SQL 漏拦截、标准题答案变化或 P95 明显恶化时，不得只凭少量手工演示合并。

## 报告格式

公开报告只包含：

- 版本和环境；
- 汇总指标；
- 按难度、模式和路由的聚合结果；
- 脱敏后的失败原因；
- 已知限制和后续动作。

内部受控报告可以包含题号、SQL、标准结果和详细错误，但必须放在 Git 忽略目录，访问权限与官方数据一致。

## 责任边界

- 05号业务语义负责人：表达映射、衍生与复合口径、题库异常审核；
- 数据负责人：数据版本、字段映射和查询结果对账；
- 测试负责人：评测执行、答案比对、回归报告和发布建议；
- 项目负责人：评测接口、版本冻结和最终发布门槛；
- 产品与前端负责人：根据已验证能力选择推荐问题和展示，不接触测试答案。
