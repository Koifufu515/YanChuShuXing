# 03号任务：前端实现与产品集成

> 对应 GitHub Issue：#3“前端实现与产品集成”
> 负责人：项目负责人和两位02号产品队友，项目负责人承担主要责任
> 当前状态：任务说明已确认，等待02号产品原型及正式接口契约进入实施。

第一版需要跑通以下完整流程：

```text
输入与官方200道问题同类型的自然语言问题
  -> 后端完成真实查询
  -> 前端按题型展示准确结果
  -> 用户保存分析记录
  -> 刷新或重新进入页面后查看记录
  -> 修改记录标题和人工备注
```

第一版以单轮问数准确性为核心，同时提供轻量分析记录功能。上下文传递、继续追问和多轮分析进入后续版本规划。现有Streamlit Demo、API Client、安全检查、技术详情和稳定性处理作为实施基线保留。

## 一、项目负责人的任务

项目负责人负责公共技术模块和最终集成。

### 接口与现有工程审计

首先检查当前仓库中的前端结构、查询接口、后端响应、错误状态和已有组件，形成统一的前端接口字段表。字段至少覆盖：

| 字段类别 | 需要明确的内容 |
|---|---|
| 问题信息 | 原始问题、请求编号、查询时间 |
| 执行信息 | 执行状态、结果类型、耗时 |
| 业务结果 | 业务结论、结构化数据 |
| 业务口径 | 机构、指标、日期或时间范围、单位 |
| 异常状态 | 错误类型、用户提示、需要补充的条件 |

项目负责人为单值、跨期比较、排名、趋势和综合分析五类结果分别准备一份标准返回样例，供公共结果组件、主页面和分析记录详情共同使用。

### 公共API与结果转换模块

建立统一API调用模块和结果转换模块：

- API调用模块负责提交问题、接收响应、处理加载状态、超时和结构化错误；
- 结果转换模块负责将后端响应转换为五类结果组件可直接读取的统一前端数据结构；
- 主页面和分析记录详情通过同一结果结构调用公共组件，保证相同结果的展示一致。

### 分析记录公共模块

确定分析记录数据结构，并整理以下后端接口契约：

1. 创建分析记录；
2. 获取分析记录列表；
3. 获取分析记录详情；
4. 修改分析标题；
5. 修改人工备注。

分析记录至少保存记录编号、原始问题、结果类型、业务结论、结构化数据、机构、指标、日期、单位、查询时间、标题和人工备注。具体存储结构、页面路由和接口路径由项目负责人统一维护。

### 集成与交付

两位队友完成页面后，项目负责人完成接口接入、代码审核、冲突处理、真实后端联调和最终合并。

项目负责人最终交付：

1. 前端接口字段表；
2. 五类结果标准返回样例；
3. 公共API调用模块；
4. 结果转换模块；
5. 分析记录接口与数据结构；
6. 完整集成版本。

## 二、第一位队友的任务

第一位队友负责完整实现智能问数主页面。

### 页面结构

主页面包含：

- 问题输入框；
- 代表性示例问题；
- 提交按钮；
- 查询状态区域；
- 查询结果区域；
- 保存分析入口。

示例问题点击后填入输入框，由用户确认后提交。提交过程中展示加载状态并控制重复请求。

### 查询状态

页面覆盖以下主要状态：

| 状态 | 页面行为 |
|---|---|
| 正常结果 | 调用对应公共结果组件展示结论、数据和图表 |
| 条件不足 | 提示缺少的机构、指标或日期，并保留原问题 |
| 无数据 | 展示当前查询条件，提供调整条件后的重新查询入口 |
| 查询失败 | 保留原问题，展示可理解错误并提供重新提交入口 |

查询成功后，页面提供“保存分析”按钮，通过项目负责人提供的接口保存当前问题和查询结果。

第一位队友最终交付：

1. 能够调用真实后端的智能问数主页面；
2. 主要查询状态与重复提交控制；
3. 五类公共结果组件接入；
4. 分析记录保存入口；
5. 页面截图和基本运行测试记录。

## 三、第二位队友的任务

第二位队友负责五类公共结果组件和轻量分析记录页面。

### 五类公共结果组件

| 组件 | 展示内容 |
|---|---|
| 单值组件 | 业务结论、核心数值、单位、机构、指标和日期 |
| 跨期比较组件 | 当前值、基准值、变化量和变化率 |
| 排名组件 | 名次、机构、指标值和单位，并根据数据情况展示柱状图 |
| 趋势组件 | 折线图和日期明细 |
| 综合分析组件 | 总体结论、关键指标和结构化数据表 |

五类组件统一读取项目负责人确定的数据结构，并同时用于智能问数页面和分析记录详情。

### 分析记录页面

分析记录页面包括记录列表和记录详情：

- 记录列表展示标题、原始问题、查询时间、主要机构和指标；
- 记录详情展示完整结论、结构化数据、图表和查询口径；
- 详情页支持修改分析标题和人工备注；
- 记录详情复用五类公共结果组件，保持历史结果与首次查询展示一致。

第二位队友最终交付：

1. 五类公共结果组件；
2. 分析记录列表；
3. 分析记录详情；
4. 标题和备注编辑功能；
5. 页面截图和基本运行测试记录。

## 四、开发顺序与协作方式

项目负责人先完成接口审计、统一字段、五类返回样例和前端目录骨架。公共契约确认后，两位队友并行开发：

- 第一位队友维护智能问数页面及其页面样式；
- 第二位队友维护公共结果组件和分析记录页面；
- 项目负责人维护公共API、结果模型、分析记录接口、数据库结构和页面路由，并负责最终集成。

三个人分别在独立分支开发，通过Pull Request合并。涉及公共接口、数据结构、页面路由和分析记录存储的修改，由项目负责人审核后进入主分支。

建议实施顺序：

1. 项目负责人完成接口与现有工程审计；
2. 冻结前端字段表、五类样例和分析记录契约；
3. 建立前端目录骨架与公共模块入口；
4. 两位队友并行开发主页面和结果/记录页面；
5. 项目负责人接入真实后端并完成联调；
6. 三人共同执行五类训练题与分析记录闭环验收；
7. 整理截图、测试记录和已知问题后提交最终PR。

## 五、验收标准

第一版需要完成完整链路：用户输入问题，前端提交真实后端，后端返回查询结果，页面根据题型展示对应组件，用户保存分析记录，刷新或重新进入页面后仍能打开该记录，并能够修改标题和备注。

验收时从训练题中分别选择单值、跨期比较、排名、趋势和综合分析问题，检查：

1. 问题提交与查询状态；
2. 后端接口状态与结构化响应；
3. 业务结论、数据、单位和日期；
4. 表格与图表展示；
5. 条件不足、无数据和查询失败状态；
6. 分析记录保存与刷新后回看；
7. 标题和备注修改；
8. 同一结果在主页面和记录详情中的展示一致性。

最终提交前端联调记录，列明测试问题、接口状态、页面结果和发现的问题。

3号最终形成三类成果：

1. 项目负责人交付公共技术模块和集成版本；
2. 第一位队友交付智能问数主页面；
3. 第二位队友交付结果组件和分析记录页面。
