# 言出数行数据库与指标字典

> 状态：正式数据审计模板。字段名、类型、主键和关系必须由 Issue #4 根据官方文件核对后填写；本文不提前编造正式 Schema。

## 一、数据环境

项目采用两套相互隔离的数据环境：

| 环境 | 用途 | 数据来源 | 是否可公开 |
|---|---|---|---|
| Demo | 开发、CI、公开演示、迁移回归 | 确定性模拟数据 | 可以 |
| Real | 正式业务验证和比赛评测 | 五张官方表 | 原始数据和答案不公开 |

前端始终通过后端 API 查询，不直接读取 Excel 或连接数据库。Demo 和 Real 必须使用独立数据库文件、配置和初始化入口，禁止用官方数据覆盖 `data/processed/bankinsight.db`。

## 三、正式建模待决策项

以下内容必须由数据审计和查询样例共同决定：

1. 五张源表是原样落地、规范化拆分还是分层建模；
2. 机构、指标和日期的稳定业务键；
3. 指标数值的存储类型、单位和精度；
4. 衍生规则是配置、SQL 模板还是独立规则表；
5. Real 数据库文件、迁移版本和初始化入口；
6. 常用机构、指标、日期组合的索引；
7. 正式查询与答案标准化所需的 SQLite 兼容函数和时间处理规则；
8. 数据脱敏、访问范围和评测结果输出边界。

每个决策必须记录数据证据、影响范围、负责人和审核人。未经项目负责人和业务负责人确认，不修改公共 Schema 与 Metrics 配置。

## 四、数据质量验收

Issue #4 至少应输出下列检查结果：

| 检查项 | 验收内容 |
|---|---|
| 文件完整性 | 五张表均可读取，Sheet 名、行列数和校验值记录完整 |
| 类型 | 日期、数值、文本和枚举可稳定转换 |
| 唯一性 | 候选主键无未解释重复 |
| 关联性 | 事实表机构和指标可关联到维表 |
| 缺失 | 缺失率、业务含义和处理策略明确 |
| 异常 | 极值、负值、单位和跨期跳变有核对记录 |
| 对账 | 源文件与目标数据库行数、关键汇总和抽样值一致 |
| 可重复 | ETL 重复执行不产生重复记录或不可控状态 |
| 安全 | 原始数据、答案和敏感结果未进入公开 Git |

## 五、指标字典模板

正式指标应按下列字段登记：

| 字段 | 说明 |
|---|---|
| 指标 ID | 稳定英文标识 |
| 中文名称 | 官方名称 |
| 业务定义 | 面向用户的口径说明 |
| 计算公式 | 基础取值或衍生计算 |
| 单位与精度 | 原始单位、展示单位和舍入规则 |
| 时间口径 | 日期字段、期点/期间、比较基准 |
| 机构口径 | 可用机构层级与聚合方式 |
| 依赖字段 | 来源表和字段 |
| 默认过滤 | 状态、范围和空值规则 |
| 可用维度 | 可分组和筛选维度 |
| 支持状态 | 待审计、已映射、已验证、已上线 |
| 审核人 | 业务、数据和项目负责人 |

官方指标清单和衍生维度说明是正式口径来源。团队只做规范化映射、实现和验证，不重新发明一套与官方数据脱节的指标体系。
