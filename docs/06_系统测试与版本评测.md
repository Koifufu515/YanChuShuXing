# 06号负责人：系统测试与版本评测

> 对应 GitHub Issue：#6“官方题库批量评测、错误归因与版本回归验收”
> 资料边界：公开仓库只保存评测程序、脱敏样例、规则说明和聚合报告；完整题目、官方答案、逐题 SQL、数据库结果与失败日志保存在 Git 忽略的受控目录。

## 一、角色定位

06号负责人负责搭建并运行官方题库批量评测，制定结果标准化与自动判分规则，完成失败问题的第一层归因，维护专项测试用例，并为重要版本提供回归差异和发布建议。归因后，业务语义问题提交05号复核，数据和字段问题提交04号处理，Prompt、模型、SQL生成、接口和结果格式化问题提交项目负责人及对应开发成员处理；修复完成后由06号执行复测。

系统正确性以“实际查询结果经过标准化后与《问题答案清单》的官方答案比较”为准；查询过程通过语义识别、实际生成 SQL、安全状态、数据库结果和错误日志进行排查。

## 二、官方题库批量评测

评测程序逐题读取120道训练题、40道验证题和40道测试题，将问题作为真实用户输入提交给后端查询接口，使其完整经过：

```text
自然语言问题
  -> 语义理解
  -> SQL 生成
  -> SQL Safety
  -> 正式只读数据库执行
  -> 结果格式化
  -> 答案标准化
  -> 官方答案比较
```

每道题至少保存以下字段：

- 问题编号和数据分区；
- 代码版本、配置版本、模型和数据库版本；
- 语义识别结果，包括机构、指标、时间、比较方式和业务公式；
- 实际路由、Rule 命中状态与实际生成 SQL；
- SQL Safety 状态；
- 数据库原始结果、返回行数和执行状态；
- 系统最终答案与规范化后的答案；
- 官方答案比对结果；
- 端到端耗时和各阶段可获得的耗时；
- 失败阶段、错误类型和脱敏错误信息。

06号通过批量程序运行、检查和汇总200道题。程序调用真实后端接口或与线上一致的应用入口，确保每道题完整经过Pipeline。

## 三、结果标准化与自动判分

系统结果完成标准化后直接与官方答案比较。标准化规则采用版本化管理并经过人工抽样，包括：

- 金额单位统一，例如元、万元、亿元；
- 百分比、小数与百分点区分，以及统一精度和舍入；
- 日期格式和时间范围表达统一；
- 机构名称、机构编号和允许简称映射；
- 排名题的机构、名次、并列关系与结果值核对；
- 列表和集合题的顺序要求；
- 空值、无数据、零值和缺失条件的区别；
- 综合题所要求机构、指标、方向与结论是否完整。

单值题比较数值与单位；排名题比较机构、名次和结果；综合问题检查官方答案要求的关键内容。宽松规则不得掩盖错误，也不得通过忽略关键字段虚增准确率。

05号确认的异常题通过题库异常清单单独标记和统计。项目负责人决定继续按官方答案判分、列为异常题，还是从某项主指标中排除；06号按照确认后的策略执行评测。

## 四、失败问题第一层归因

06号依据完整链路日志，将失败至少分为：

| 失败类型 | 判断依据 | 交接对象 |
|---|---|---|
| 语义理解错误 | 机构、指标、时间、比较方式或业务公式识别错误 | 05号复核业务口径；技术实现交项目负责人或开发成员 |
| SQL 生成错误 | 字段、连接、聚合、筛选、排序或时间条件错误 | 项目负责人或生成模块开发成员 |
| Safety 拒绝 | 合法查询被误拒绝，或危险查询未拦截 | 安全模块开发成员、项目负责人 |
| 数据库执行错误 | 表字段不存在、类型错误、执行异常或超时 | 04号数据负责人及相应开发成员 |
| 结果格式化错误 | 原始结果正确，但单位、排序、摘要或结构错误 | 格式化模块开发成员 |
| 前端展示错误 | API 结果正确，但页面丢失、错位或状态异常 | 前端负责人 |
| 官方题库异常 | 问题表述、衍生规则、事实数据与答案冲突 | 05号复核，项目负责人决定处理策略 |

若最终结果错误，06号检查语义、实际 SQL、安全状态、数据库原始结果、最终答案和错误日志，形成可复现证据后分派。修复后重跑原问题，同时检查其他问题是否出现退化。

## 五、固定专项测试用例库

除200道官方题库外，06号维护独立于官方答案的专项用例，至少覆盖：

| 场景 | 预期行为 |
|---|---|
| 空问题 | 返回结构化参数错误，服务不崩溃 |
| 不存在的机构 | 返回无匹配机构或需要修正条件 |
| 超出数据日期范围 | 返回无数据或明确日期范围错误 |
| 危险数据库操作 | Safety 拒绝，不调用数据库执行器 |
| 接口或模型超时 | 返回结构化超时错误，可继续查询 |
| 无数据返回 | 区分无数据、零值与系统失败 |
| 前端错误状态 | 结束加载状态，不显示内部异常，可再次查询 |

每条用例记录输入、预期行为、实际行为、是否通过、运行版本和证据。另选少量代表性题目重复运行，检查同一版本在相同数据和配置下能否返回一致业务结果。

## 六、数据分区使用规则

| 分区 | 用途 | 禁止事项 |
|---|---|---|
| 训练集（120题） | 发现问题，修改 Prompt、业务配置、Schema 说明和结果处理逻辑，完成内部回归 | 不得把答案硬编码进程序 |
| 验证集（40题） | 比较不同 Prompt、配置或系统方案在未参与修改的问题上的表现 | 不得写入 Prompt、Rule、Few-shot 或推荐问题 |
| 测试集（40题） | 阶段验收和最终评估 | 不得根据测试答案反复调参，不得泄漏到开发代码或公开报告 |

测试集内容不得进入 Rule、Prompt、Few-shot 示例、前端推荐问题或开发配置。训练集内部若建立开发子集，必须标明这是项目内部策略，不得冒充官方分区。

## 七、版本回归与发布建议

第一次正式全量评测保存为基线。以后 Prompt、业务配置、Schema、数据库、路由、Formatter 或代码发生变化时，06号重新运行同一套验证题和专项用例，报告至少列出：

- 上一版错误、本版正确的问题；
- 上一版正确、本版错误的退化问题；
- 新增接口失败、Safety 异常或超时；
- 各分区、难度和问题模式的准确率变化；
- SQL 执行成功率、端到端成功率和P50/P95耗时变化；
- 已知异常题对统计结果的影响；
- 建议合并、阻塞或回退的结论。

06号提供证据、失败清单和发布建议，项目负责人据此作出主分支合并决定。

## 八、四项正式交付物

1. **官方题库批量评测程序**：读取受控题库、调用后端、保存全链路记录并生成汇总。
2. **结果标准化与自动判分规则**：覆盖数值、单位、百分比、日期、机构、排名、集合和综合答案。
3. **固定专项测试用例库**：覆盖输入边界、安全、超时、无数据、恢复和前端状态。
4. **版本评测结果与回退清单**：保存基线、版本差异、失败归因、退化项和发布建议。

建议目录如下，正式实现时可由技术成员按现有测试结构微调：

```text
evaluation/
├── runner.py
├── normalization.py
├── scoring.py
├── models.py
└── reports/

tests/evaluation/
├── test_normalization.py
├── test_scoring.py
└── test_special_cases.py

data/private/evaluation/
├── official_questions.*
├── official_answers.*
├── run_details/
└── version_reports/
```

公开仓库保存程序、规则、脱敏样例和聚合报告；完整题库、官方答案、逐题SQL、真实数据库结果和敏感日志保存在受控目录。

## 九、协作边界

- **与05号**：接收已确认的业务口径、表达映射和异常清单；业务理解疑点交05号复核。
- **与04号**：核对数据库版本、字段映射、数据完整性和执行问题，并提交数据层问题证据。
- **与项目负责人及开发成员**：提交 Prompt、模型、SQL生成、Safety、接口和Formatter问题的复现证据；开发成员完成修复后由06号复测。
- **与产品和前端负责人**：提供前端状态与业务展示的失败证据；官方测试答案与推荐问题、页面内容保持隔离。

## 十、验收标准

Issue #6只有在以下条件全部满足后才能关闭：

1. 200道题可以按训练、验证、测试分区自动运行；
2. 每题保留语义、实际 SQL、Safety、数据库结果、最终答案、耗时和错误信息；
3. 标准化规则覆盖金额、百分比、精度、日期、机构、排名和空值；
4. 系统结果直接与官方答案比较，异常题能够单列；
5. 失败问题可按链路阶段完成第一层归因并交接；
6. 专项测试覆盖边界、安全、超时、无数据和错误恢复；
7. 已建立首个版本基线，并能输出新旧版本改善与退化清单；
8. 测试集与Prompt、Rule、Few-shot、推荐问题和开发代码保持隔离；
9. 公开仓库仅保存程序、规则、脱敏样例和聚合报告，受控资产存放位置清晰；
10. 评测程序、规则、专项用例和版本报告均通过审核并合并。
