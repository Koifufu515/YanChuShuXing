# Issue 草案：下钻、导出、性能与部署

> 状态：候选草案，前四项能力达到验收基线后再决定是否整体或拆分创建正式 Issue。

## 任务目标

完成比赛交付阶段的分析体验和工程验证：层级下钻、权限受控导出、可保存看板、基于正式评测的性能优化、Docker/Linux 交付以及标准外部接口适配示例。

## 依赖条件

- 真实指标、机构层级、图表、权限和审计能力稳定；
- 官方200题评测能提供 P50/P95、失败率和慢查询分类；
- 产品方案明确下钻路径、导出范围和看板使用场景；
- 比赛平台和目标部署环境已确认。

## 主要交付物

- 机构、时间和指标层级下钻及筛选状态传递；
- CSV/XLSX 导出、保存视图或轻量经营看板；
- 索引、查询计划、上下文大小、缓存和超时优化报告；
- Dockerfile、Linux 部署验证、健康检查、资源说明和提交包；
- 外部数据库、身份平台或模型网关的接口规范、Mock 与一个适配器示例。

## 涉及模块

`frontend/`、导出与看板服务、数据库索引、部署文件、外部适配器、性能测试和交付文档。是否拆分为多个 Issue 由项目负责人根据剩余赛程决定。

## 验收指标

下钻和导出不改变指标口径且遵守权限；导出结果与页面一致；Rule 和本地数据库路径以 P95 不高于3秒为目标，LLM 路径单独报告；干净 Linux/Docker 环境能按文档启动并完成核心查询；外部适配只宣称已验证的 Mock 或实际联调范围。

## 负责人角色

产品负责人定义体验，前端负责人实现页面，数据负责人负责索引、导出和数据适配，业务负责人核验指标，测试负责人执行性能与部署验收，项目负责人负责架构、发布和接口审核。

## 不能提前开展的条件

权限、审计、正式数据和性能基线未稳定时不得开放导出或做无目标优化；没有目标环境和联调证据时不得宣称国产化或真实银行系统对接完成。
