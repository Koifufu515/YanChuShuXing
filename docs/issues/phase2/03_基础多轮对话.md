# Issue 草案：基础多轮对话

> 状态：候选草案，单轮真实数据查询和图表解释稳定前不得创建正式 GitHub Issue。

## 任务目标

在现有 `conversation_id` 基础上实现最小、可测试的连续问数，包括历史存储、最近上下文召回、条件继承与覆盖、澄清和会话清理。首版不引入 Agent、LangGraph 或任务规划器。

## 依赖条件

- 单轮复杂问题在正式评测中达到团队确认的稳定基线；
- 产品负责人已给出连续追问、澄清、清空和失败恢复流程；
- 会话数据的保留、隔离与脱敏规则已经评审；
- API 向后兼容方案已经冻结。

## 主要交付物

- 会话与消息存储设计及适配器；
- 有限窗口或摘要式上下文召回；
- 时间、机构、指标、排序和限制条件的继承与覆盖规则；
- 会话界面、清理入口、隔离和连续追问测试。

## 涉及模块

会话存储 Port/Adapter、上下文解析、API 可选字段、`frontend/` 和测试。不得把会话数据库访问直接写入 Pipeline。

## 验收指标

预定义连续追问能够正确继承、覆盖和清除条件；不同用户和会话互不串线；上下文大小和保留期限受控；历史不可用时返回可理解错误且不执行错误 SQL；原有单轮请求保持兼容。

## 负责人角色

产品负责人定义交互，数据负责人设计存储，业务负责人维护上下文规则，前端负责人实现会话体验，测试负责人负责隔离和回归，项目负责人审核接口和编排。

## 不能提前开展的条件

单轮评测未稳定、上下文安全边界未确认或团队仍需要修改核心数据模型时，不得开始；不得用多 Agent 掩盖单轮 NL2SQL 的基础错误。
