# Issue 草案：权限与持久化审计

> 状态：候选草案，Real Schema、用户范围和敏感级别冻结前不得创建正式 GitHub Issue。

## 任务目标

将现有表字段 Safety 和 Audit Port 扩展为最小可用的银行查询治理能力：按用户角色和机构范围限制查询，对敏感字段执行拒绝或脱敏，并持久化关键查询事件。

## 依赖条件

- Real Schema 已确定机构层级、字段敏感级别和数据归属；
- 团队已确定最小用户角色和权限矩阵；
- 审计保留、脱敏与访问规则已经评审；
- 项目负责人已冻结身份上下文到 `UserContext` 的接口。

## 主要交付物

- 用户角色、机构范围、字段权限和脱敏策略；
- 身份上下文适配器与动态 `UserContext`；
- 持久化 Audit Adapter、审计表、保留策略和最小查询入口；
- 越权、脱敏、绕过、审计完整性和敏感日志测试。

## 涉及模块

权限与审计数据表、Safety Adapter、Audit Adapter、Composition Root、API 身份上下文和测试。Pipeline 继续只调用 Port，不直接实现权限或持久化。

## 验收指标

越权机构、拒绝字段和敏感数据查询被稳定拒绝或脱敏；Safety 拒绝后不访问数据库；请求开始、成功、拒绝和失败均有审计记录；日志不包含密钥、高敏原值或第三方原始异常。

## 负责人角色

数据负责人负责模型和适配器，测试负责人负责安全验收，项目负责人维护公共契约；产品、前端和业务负责人分别提供提示、入口和敏感级别规则。

## 不能提前开展的条件

机构层级、字段敏感级别、角色矩阵或审计数据边界未确认时不得编码；不得把静态允许表列表包装成“完整权限系统”。
