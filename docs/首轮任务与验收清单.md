## 推荐执行顺序

项目负责人先完成仓库成员邀请、Issue 分配和主分支保护。产品方案、数据库差距分析、业务问题指标库和测试基线随后可以并行开始，其中 #4 与 #5 需要持续互相校验。#3 不应在 #2 的核心流程和优先级尚未评审时大规模开发，但可以先完成现状盘点和技术风险清单。#6 从当前主干建立基线，并参与之后每一个 Pull Request，而不是等全部开发完成后再测试。

## 共同验收关卡

任何 Issue 都不能因为上传一个文件或提交一次代码就关闭。负责人需要确认约定交付物完整，相关角色已经审核，必要测试与手工检查通过，README、CHANGELOG 或领域文档已经同步，敏感信息检查无异常，Pull Request 已合并到 `main`，并且后续接力人已收到明确输入。只完成分析而尚未进入开发时，应记录“本阶段分析完成”和后续接力 Issue，不能把规划能力写成已经实现。

## 项目负责人检查

项目负责人每周至少检查一次六个 Issue 的负责人、依赖、阻塞、PR 和验收证据。涉及 `application/`、`ports/`、`bootstrap/container.py`、数据库结构、`schema.yml`、`metrics.yml`、README、CHANGELOG 或公共接口的修改，必须按 CODEOWNERS 和协作规范追加项目负责人审核。
