## 任务1：建立团队协作规则与核心文件审核机制

**GitHub Issue**：[#1 建立团队协作规则与核心文件审核机制](https://github.com/Koifufu515/BankInsight/issues/1)

**任务目标**：把角色、分支、PR、审核、合并和核心文件所有权正式落到仓库与 GitHub 设置中。

**具体交付物**：团队规范、首轮任务清单、四类 Issue 模板、PR 模板、CODEOWNERS、更新后的贡献指南；创建六个首轮 Issue；邀请成员后完成 `main` 保护设置。

**建议修改文件**：`docs/团队分工与协作规范.md`、`docs/首轮任务与验收清单.md`、`.github/`、`CONTRIBUTING.md`、README 和版本记录。

**验收标准**：

- GitHub 能识别四类 Issue 表单与 PR 模板；
- CODEOWNERS 路径均在仓库真实存在，且只覆盖约定核心路径；
- 六个 Issue 均包含交付物、验收标准、依赖和负责人角色；
- README 可进入全部协作文档；
- `main` 保护规则已配置，或明确记录未配置原因和手动步骤；
- 本次变更未触碰业务逻辑。

**依赖关系**：无。完成后其余任务才能正式认领。

## 任务2：提交产品功能与交互优化方案

**GitHub Issue**：[#2 提交产品功能与交互优化方案](https://github.com/Koifufu515/BankInsight/issues/2)

**任务目标**：以用户任务和比赛演示为依据，确定下一轮页面优化优先级，避免先写代码后补产品理由。

**具体交付物**：`docs/product/产品功能与交互优化方案.md`，包含目标用户、核心场景、现状问题、优先级、页面流程、线框图或原型、推荐问题、加载/空/错误/成功状态、图表需求、每项功能验收标准，以及桌面端和移动端后续规划。

**建议修改文件**：仅新增或更新 `docs/product/` 下的产品文档和原型资源；首轮不改后端 API。

**验收标准**：

- 至少覆盖一次完整问数流程及四类页面状态；
- 每项优化均说明用户问题、优先级和可观察验收标准；
- 明确哪些能力已有后端支持、哪些只是产品规划；
- 前端负责人确认可实现，业务负责人确认表述与场景合理；
- 不把未实现能力写成当前已完成能力。

**依赖关系**：依赖任务1；任务3在本方案评审后实施。

## 任务3：提交前端技术改造清单并完成首批低风险优化

**GitHub Issue**：[#3 提交前端技术改造清单并完成首批低风险优化](https://github.com/Koifufu515/BankInsight/issues/3)

**任务目标**：把评审通过的产品方案转换为可维护的 Streamlit 改造项，并完成一批不改后端契约的低风险优化。

**具体交付物**：`docs/frontend/前端技术改造清单.md`；一组范围明确的前端改动；桌面和窄屏截图；加载、成功、错误及恢复路径的验证记录。

**建议修改文件**：`frontend/`、`.streamlit/`、相关前端测试和 `docs/assets/`。不得修改 Pipeline、Generator、Safety、Schema 或 API 契约。

**验收标准**：

- 技术清单逐项对应已评审产品需求；
- 页面数据全部来自现有 API 或只读数据层，不写死查询结果；
- 三个标准问题、一个不支持问题和失败后恢复均可用；
- 桌面及窄屏无明显溢出、错位或状态残留；
- 自动化回归通过，产品负责人和测试负责人完成审核。

**依赖关系**：依赖任务2方案评审；可与任务4、5并行。

## 任务4：完成数据库与真实数据接入差距分析

**GitHub Issue**：[#4 完成数据库与真实数据接入差距分析](https://github.com/Koifufu515/BankInsight/issues/4)

**任务目标**：说明当前模拟数据库能回答什么、缺什么，以及真实数据到来时如何替换，避免未经口径讨论直接改表。

**具体交付物**：`docs/data/现有数据库与真实数据接入差距清单.md`，包括现有10表及关系、数据量和质量、已支持问题、缺失字段、待新增数据主题、真实数据字段映射、脱敏与质量规则、替换范围和优先级。

**建议修改文件**：首轮以 `docs/data/` 分析文档和只读校验脚本为主；不得直接修改 `sql/schema.sql`、Schema 或指标口径。

**验收标准**：

- 逐表核对 DDL、Schema YAML 和演示数据库，不根据文件名猜测；
- 每个缺口关联至少一个业务问题或指标；
- 区分“缺字段”“缺数据”“缺口径”和“缺权限”四类问题；
- 给出真实数据接入后的替换与兼容方案；
- 业务负责人和测试负责人完成审核。

**依赖关系**：依赖任务1；与任务5互相校验。

## 任务5：整理首批业务问题库与指标口径

**GitHub Issue**：[#5 整理首批业务问题库与指标口径](https://github.com/Koifufu515/BankInsight/issues/5)

**任务目标**：形成全组唯一的候选问题与指标事实表，为 Rule、LLM、数据和展示提供共同依据。

**具体交付物**：`docs/business/业务问题库与指标口径.md`，整理15至20个问题。每项包含业务场景、用户问法、指标中文名、英文标识、公式、时间口径、表字段、展示形式、建议路由、Gold SQL、预期结果和数据准备状态。

**建议修改文件**：首轮主要新增 `docs/business/` 文档；只有口径经过评审后，后续 PR 才可改 `config/metrics.yml`。

**验收标准**：

- 候选问题总数为15至20个，覆盖当前六类业务场景中的合理范围；
- 明确标记“已支持”“适合新增 Rule”“LLM 试验”“等待真实数据”；
- 公式、时间边界、空值和分母为零处理清楚；
- Gold SQL 与当前表字段一致，无法执行的项目明确说明数据依赖；
- 数据负责人核验可计算性，项目负责人审核公共口径。

**依赖关系**：依赖任务1；与任务4互相校验，可与任务2并行。

## 任务6：建立功能验收模板并完成当前版本基线测试

**GitHub Issue**：[#6 建立功能验收模板并完成当前版本基线测试](https://github.com/Koifufu515/BankInsight/issues/6)

**任务目标**：建立所有后续 PR 共用的验收方式，并记录当前版本可重复的质量基线。

**具体交付物**：`docs/testing/功能验收与测试模板.md`、`docs/testing/当前版本基线测试报告.md`，必要时增加不改变业务逻辑的测试辅助脚本。

**建议修改文件**：`tests/`、`docs/testing/`；仅在发现明确缺陷时另开 Issue，不把大范围修复混入该 PR。

**验收标准**：

- 模板覆盖功能目标、接口、Rule、LLM、Safety、Metadata、数据库约束、页面状态、文档和敏感信息检查；
- 基线报告记录环境、命令、87项现有测试结果、数据库初始化、三条标准查询、结构化错误和已知限制；
- 失败项具备复现步骤、严重级别和建议负责人；
- 报告不包含 API Key、第三方原始异常或真实客户数据；
- 项目负责人完成审核。

**依赖关系**：依赖任务1和当前稳定主干；为后续所有功能 PR 提供验收基线。

## 首轮共同完成标准

每个 Issue 均需由一名具体成员认领，使用独立分支和独立 PR，按交叉审核矩阵完成审核。未合并的成果不得在 README 或答辩材料中标记为已完成。
