# 言出数行正式赛题阶段任务计划

> 状态管理以本文件、GitHub Issues 和 Pull Request 为准。Demo 开发历史已由 `demo-baseline` Tag、Release、Git 提交与 CHANGELOG 保存，不再在当前计划中保留逐 Sprint 流水账。

## 阶段总览

| 阶段 | 目标 | 负责人 | 状态 | 关键依赖 |
|---|---|---|---|---|
| 1 | 仓库与任务重构 | 项目负责人 | 进行中 | Demo 基线可回退 |
| 2 | 五张官方表审计和建库 | 数据负责人 | 待开始 | 阶段1 |
| 3 | 官方问题分类与 Gold SQL | 业务负责人 | 待开始 | 阶段2字段字典 |
| 4 | 正式查询链路迁移 | 项目负责人、数据负责人、业务负责人 | 待开始 | 阶段2、3 |
| 5 | 产品与前端优化 | 产品负责人、前端实现负责人 | 待开始 | 阶段2、3的稳定契约 |
| 6 | 回归评测 | 测试负责人 | 待开始 | 阶段4、5 |
| 7 | 答辩版本收口 | 全组，项目负责人集成 | 待开始 | 阶段6通过 |

## 阶段 1：仓库与任务重构

**目标**

将项目从模拟数据原型切换为正式赛题工作方式，同时保留可靠的 Demo 回退点。

**负责人**

项目负责人。

**输入**

- 当前可运行 Demo；
- 六个现有 GitHub Issue；
- 五张官方表和赛事资料；
- 当前文档、脚本和测试。

**交付物**

- `demo-baseline` Tag 与 Release；
- 统一品牌和仓库名称；
- 更新后的 Issue #1 至 #6；
- 精简后的 README、正式文档、协作规范和安全规则；
- 可通过测试的独立 Pull Request。

**依赖关系**

无。该阶段必须先完成，后续成员才从稳定主线开始工作。

**验收标准**

- Demo 可从 Tag 恢复；
- 正式文档不再把模拟数据写成比赛数据；
- 失效链接和一次性工作底稿已清理；
- 核心代码、Demo 数据、接口和测试未被破坏；
- PR 通过审核后合并。

## 阶段 2：五张官方表审计和建库

**目标**

审计机构信息、指标清单、衍生维度、指标数据和问题答案清单，形成规范化、可重复、可验证的数据底座。

**负责人**

数据库与数据负责人；项目负责人和业务负责人审核公共结构。

**输入**

- 五张官方表原件；
- 赛事数据使用规则；
- Demo 数据库及现有 Executor 接口。

**交付物**

- 官方数据审计报告；
- 数据字典、候选主键和表关系；
- 字段映射、类型转换、缺失重复和异常处理规则；
- Demo/Real 双环境方案；
- 可重复执行、无数据泄漏的 ETL；
- 约束、索引和数据量对账结果。

**依赖关系**

依赖阶段1；其字段字典是阶段3、4、5的输入。

**验收标准**

- 五张表记录数和关键字段完成对账；
- 主键、外键候选和时间口径有证据；
- 问题答案与业务事实隔离；
- 官方原始数据未进入公开仓库；
- ETL 可重复运行且不产生重复数据；
- 公共 Schema 修改通过项目负责人 Review。

## 阶段 3：官方问题分类与 Gold SQL

**目标**

把官方问题整理为可维护的问题模式、指标语义和评测基准，不通过硬编码泄漏答案。

**负责人**

业务问题库与指标负责人；数据负责人核对字段，项目负责人审核公共配置。

**输入**

- 官方问题答案清单；
- 阶段2数据字典；
- 官方指标与衍生维度说明。

**交付物**

- 问题分类与难度分布；
- 机构、指标、时间、比较、筛选和输出形式的语义映射；
- 训练分区 Gold SQL 与审核记录；
- Rule、LLM 和暂不支持问题的路由建议；
- 推荐问题候选和预期展示形式；
- 更新后的正式指标与 Schema 说明。

**依赖关系**

依赖阶段2字段契约；与阶段2后半段可在接口冻结后并行。

**验收标准**

- 每个 Gold SQL 可执行并与标准结果核对；
- 指标公式、时间基准和单位经过业务审核；
- 测试分区内容不进入 Rule、Prompt、推荐问题或开发代码；
- 公共配置变更通过项目负责人 Review。

## 阶段 4：正式查询链路迁移

**目标**

在不推翻现有 Ports & Adapters 架构的前提下，将五张官方表和正式语义接入 Query Pipeline。

**负责人**

项目负责人负责集成；数据负责人负责 Database Adapter；业务负责人负责 Context 与 Gold Rule。

**输入**

- 阶段2正式数据库；
- 阶段3语义映射与 Gold SQL；
- 当前 Context、Generator、Safety、Executor 和 Formatter 接口。

**交付物**

- 可切换的 Demo/Real 配置；
- 正式 Context Resolver 和数据库执行适配；
- 首批验证 Rule 与 LLM 上下文；
- 统一错误、Metadata 和业务解释；
- 接口兼容性与迁移测试。

**依赖关系**

依赖阶段2、3；不得在字段和指标口径未冻结前大改 Pipeline。

**验收标准**

- 前端仍只调用 `/api/v1/query`；
- Pipeline 仍只负责编排；
- Rule 与 LLM 都经过同一 Safety；
- Demo 与 Real 环境可独立启动和回归；
- 查询结果不泄漏标准答案或敏感字段。

## 阶段 5：产品与前端优化

**目标**

依据官方数据和问题类型，把技术 Demo 改造成可用于比赛演示的经营分析产品。

**负责人**

产品与交互负责人设计；前端实现负责人和项目负责人完成页面与 API 集成。

**输入**

- 阶段2数据能力；
- 阶段3问题模式和推荐问题；
- 阶段4稳定 API。

**交付物**

- 页面信息架构和比赛演示路径；
- 机构、指标、时间和比较方式的交互；
- 业务结论、表格、适用图表和技术详情；
- 正常、条件不足、无数据、模型失败、Safety 拒绝和服务异常状态；
- 桌面端主要视口的页面截图与验收记录。

**依赖关系**

布局和状态可提前设计；业务模块、指标卡、推荐问题和图表必须等待阶段2、3确认。

**验收标准**

- 页面不直接读取 Excel 或数据库；
- 所有业务展示可追溯到官方字段或问题类型；
- 不硬编码指标结果或标准答案；
- 关键演示流程无需 Swagger 或命令行；
- 产品、业务和测试负责人共同验收。

## 阶段 6：回归评测

**目标**

建立固定题库和版本评测体系，量化每次 Rule、Prompt、Schema、路由、数据库和 Formatter 修改的影响。

**负责人**

测试、安全与文档负责人；业务负责人审核答案口径，项目负责人确认发布门槛。

**输入**

- 官方问题分区；
- Gold SQL 与标准答案；
- 正式查询链路和前端版本。

**交付物**

- 评测脚本和脱敏结果报告；
- 语义识别、路由、SQL 生成、Safety、执行和答案比对明细；
- Rule 命中率、SQL 执行成功率、答案正确率、响应时间和难度分布；
- Demo/Real 双环境回归；
- 版本差异和失败原因清单。

**依赖关系**

评测框架可在阶段2、3启动；完整端到端验收依赖阶段4、5。

**验收标准**

- 训练分区用于开发与内部回归，测试分区用于阶段验收；
- 工作簿实际存在的验证分区独立管理，不写入开发规则；
- 同一基准可重复运行并比较版本差异；
- 公开报告仅含汇总指标和脱敏失败分类；
- 核心回归通过后才允许发布。

## 阶段 7：答辩版本收口

**目标**

将数据、查询、产品、评测和文档收口为可重复演示、可解释、可回退的比赛版本。

**负责人**

全组协作，项目负责人负责最终集成和发布。

**输入**

- 阶段2至6验收结果；
- 比赛提交和智算平台要求。

**交付物**

- 冻结版本与 Release；
- 三个完整演示案例；
- 系统架构、数据治理和评测证据；
- 部署与恢复说明；
- 答辩材料和演示脚本。

**依赖关系**

阶段6通过后启动。

**验收标准**

- 新环境可按文档启动；
- 演示问题稳定返回且口径可解释；
- SQL Safety、错误处理和数据边界可现场说明；
- 评测结果可复现；
- Tag、Release、README、CHANGELOG 与代码一致。

## 范围控制

正式数据和评测基座完成前，不并行开发 RAG、向量数据库、多 Agent、复杂权限系统、桌面端、移动端或生产数据库迁移。任何新增能力必须有明确赛题价值、负责人、接口边界和验收指标，并通过独立 Issue 与 Pull Request 进入主线。
